When large tables are too expensive to run,
how should we evaluate Text-to-SQL Execution Accuracy?

When tables are too large to execute against the full production database every time—cost, latency, permissions, and security all get in the way—traditional EX breaks down. This digest surveys how academia, open source projects, and big tech companies handle evaluation in this setting.

SECTION 01

The Problem — Why large tables break EX

The standard Execution Accuracy protocol is simple: run both the predicted SQL and the gold SQL against the same database, then compare whether the result sets match. That works fine for small, clean SQLite benchmarks such as Spider 1.0, which averages fewer than 10 tables per database, but it hits a wall in enterprise scenarios.

Cost and latency

Repeatedly running queries against TB-scale BigQuery / Snowflake warehouses is slow and expensive. Gold queries in Spider 2.0 often exceed 100 lines.

Permissions and security

Production contains PII / sensitive data, and the eval pipeline may not be allowed to connect and execute directly.

Unavailable environments

Private data and cross-system workflows often cannot be reconstructed at evaluation time.

Single-database false positives / false negatives

When comparing results on fixed data, semantically different queries may coincidentally return the same result (false pass), while semantically equivalent queries may false fail because of data differences.

Concrete numbers: On fixed databases, the original Spider metric has an average false negative rate of about 2.5%, reaching 8.1% in the worst case. This is exactly the evidence that “execute and compare on one database” is not reliable.

Industry and academia have therefore converged on four main lines of work. The core idea is not to abandon execution, but to shrink the scope of execution:

SECTION 02

Academic Approaches — Top Conference Papers

2.1 Test-Suite Accuracy / Distilled Test Suites (EMNLP 2020)

Instead of comparing results on one large database, distill a small set of databases that is small but has high code coverage.

The core method: from a large number of randomly generated databases, select the smallest subset that gives the gold query high code coverage. At evaluation time, compare denotations (results) only on this distilled test suite. This yields a tight upper bound on semantic correctness while remaining highly efficient. The authors used this method to evaluate 21 models on the Spider leaderboard and manually verified that all 100 sampled cases were correct.

Use fuzzing to generate neighbor queries

Perturb the gold query—for example, > 34 vs >= 34 vs > 35—to create queries that are semantically close but not equivalent.

Generate many small random databases

Create many random databases, with the goal of finding data content that can distinguish the gold query from these neighbors.

Distill the smallest high-coverage subset

Select the smallest group of databases that gives the gold query high code coverage; this becomes the distilled test suite.

Compare denotations on the small suite

Run the predicted query only on this small suite and compare results. Execution cost drops to a very low level while false pass / false fail rates fall substantially.

Why this addresses “large tables”: You do not need to run against the original large database; you run against purpose-built small databases. This has become the official metric for Spider, SParC, and CoSQL, and covers 11 datasets including Academic, ATIS, Advising, Geography, IMDB, Restaurants, Scholar, and Yelp.

Zhong, Yu, Klein · Semantic Evaluation for Text-to-SQL with Distilled Test Suites · EMNLP 2020 · arXiv:2010.02840

2.2 Spider 2.0 (ICLR 2025 Oral) — Enterprise-grade evaluation framework

Spider 2.0 directly targets “real enterprise large databases”: 632 workflow questions from real data applications, with schemas often exceeding 1,000 columns and extreme cases reaching 3,000+ columns, running on cloud systems such as BigQuery and Snowflake. For the problem of being “too large to run casually,” it provides several important engineering choices:

  • Task settings split by execution cost: The official recommendation is to use spider2-lite / spider2-snow for benchmarking; spider2-dbt (68 tasks) is also available for a fast repository-level benchmark.
  • Oracle tables mode: The project releases ground-truth tables, letting you execute and analyze only the tables used by the correct answer (you must disclose when oracle tables are used). This is the concrete implementation of “shrinking the schema.”
  • EX remains the main metric, but because schema linking over thousands of columns is extremely hard, the best model at one point, such as o1-preview, achieved only about 17% EX, compared with 91.2% on Spider 1.0.
  • Only part of the gold answers are released; the rest are scored through leaderboard submissions to avoid test-set leakage.

Lei et al. · Spider 2.0 · ICLR 2025 Oral · arXiv:2411.07763

2.3 Execution-Guided Self-Consistency (Snowflake AI Research, 2025)

This is primarily a generation method, but its evaluation perspective is useful for large tables: judge whether two queries are the same by execution-level equivalence rather than structural comparison, and introduce exact as well as approximate execution-based similarity, with a theoretical framing based on Minimum Bayes Risk (MBR) decoding.

semantic-equiv(q1, q2) ≈ sim( exec(q1) , exec(q2) )
Key point: approximate similarity can be computed on data subsets / samples, instead of requiring exact results on the full large table

Borchmann & Wydmuch · Query and Conquer · Snowflake AI Research 2025 · arXiv:2503.24364

2.4 Approximate metrics without execution: Soft-F1 and LLM-as-Judge

Soft-F1

Give partial credit over result sets by measuring the overlap between predicted and gold results. If 99 of 100 rows are correct, Soft-F1 assigns a high score while EX assigns 0. It is useful when queries can run but results may not match perfectly, because it provides finer-grained signal and debugging evidence.

LLM-as-Judge

When execution is impossible because private data is missing or the environment is broken, use a strong LLM to compare the semantic logic of the predicted and gold SQL. It is less objective than execution, but it correlates well with human judgment.

2.5 Enterprise survey: lab numbers do not transfer cleanly to enterprises

The real-world gap: Spider leaderboard EX rose from ~40% in 2023/03 to ~76% in 2025/04; but on Spider 2.0, the best model in 2025/04 reached only about 31% EX. On Uber’s internal eval set, overlap with ground-truth tables was also only about 50%. Large schemas and semantic ambiguity are the core challenges.

Text-to-SQL for Enterprise Data Analytics · 2025 · arXiv:2507.14372

Practical composition recommendations

Small to midsize reconstructable databases

Use test-suite-sql-eval directly to distill a small test suite and push execution cost down to the minimum.

Enterprise large warehouses (BigQuery / Snowflake)

Follow Spider 2.0’s oracle tables pattern: execute only on the tables used by the gold query. For agent configuration, spider-agent-lite / snow are useful references.

Private data that cannot be executed at all

Fall back to Soft-F1 if result samples are available, or LLM-as-Judge if you can only compare semantics.

SECTION 04

Technical posts from big tech companies

Uber — QueryGPT

Uber’s SQL runs over TB-scale data every day, so its evaluation does not rely on row-by-row result-set comparison over the full database. Instead, Uber built a standardized evaluation process:

  • Human-curated golden comparison set: Select real questions from production logs, then have humans verify the correct intent, required schema, and golden SQL across multiple datasets and business domains.
  • Capture multiple signals along the generation flow across different production / staging flows, rather than looking only at whether the final result set is identical. This helps separate repeated failures from one-off defects.
  • Schema-level proxy: Use the overlap between “tables used by the generated query” and ground-truth tables (reported at roughly 50% overlap) instead of result-level comparison. This is a typical substitute when executing the whole query is impractical.

See also Uber’s Prompt Engineering Toolkit: it includes LLM-as-Judge and code-based evaluators, and sets up gatekeeping so that prompts must pass an eval threshold before going to production.

QueryGPT (2024) · Prompt Engineering Toolkit (2024)

LinkedIn — SQL Bot

LinkedIn built a custom benchmarking framework: a tailored eval set, explicit metrics, and human-plus-LLM judging to maintain accuracy, reduce hallucinations, and treat benchmarking as a continuous process integrated into product development rather than a one-off task. In practice, this avoids the cost of row-by-row comparison on production-scale tables by using humans plus LLMs to score semantics and relevance.

LinkedIn Engineering · SQL Bot (as summarized in Rincón Herrera, 2025)

Pinterest — Text-to-SQL

The main lesson from Pinterest is to evaluate table retrieval, the component that finds the right tables, in isolation. Pinterest used offline data from prior table search to first ensure that embedding-based table search was no worse than the older text-based search. In other words, “finding the right schema subset” is treated as an independently evaluable subproblem. Early evaluation of the overall Text-to-SQL system mainly aligned the implementation with published numbers by validating on Spider, rather than doing result-level EX on Pinterest’s own large data.

How we built Text-to-SQL at Pinterest (2024)

Observations from Snowflake / Cortex Analyst

Third-party evaluations point out that beyond EX, we can add Efficiency: whether the query is optimized for the warehouse architecture, including join order and use of Snowflake-specific optimizations. For a large warehouse, efficiency is itself an evaluation dimension, because a functionally correct but inefficient query can consume all available resources. This maps to the academic Valid Efficiency Score (VES), which measures execution efficiency while evaluating correctness.

SECTION 05

Decision Tree — Which approach should you use?

Can the database be reconstructed at eval time?
├─ Yes, and it's not large              → Standard EX + Test-Suite Accuracy (distill a small test suite)
├─ Yes, but very large (TB-scale cloud) → Oracle / ground-truth tables subset execution (Spider 2.0 style)
│                                          + Efficiency / VES evaluated together
└─ No (private / no environment)
     ├─ Result-set sample available     → Soft-F1 (partial credit)
     ├─ Only the query itself           → LLM-as-Judge (semantic comparison) + schema overlap proxy
     └─ Production live monitoring      → Human + LLM judges + golden set + multi-signal proxy (Uber / LinkedIn style)

One-sentence summary: When “large tables are not freely executable,” the consensus is not to “give up on execution,” but to shrink execution: shrink the data (distilled test suite), shrink the schema (oracle / relevant tables), or shrink to the semantic layer (LLM-judge / schema-overlap proxy), then pair that with human + LLM judging and golden sets for continuous production monitoring.
