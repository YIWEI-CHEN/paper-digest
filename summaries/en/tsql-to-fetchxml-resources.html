01SQL 4 CDS

Top PickOpen SourceEmbeddable via NuGetTDS endpoint
Author Mark Carrington
GitHub github.com/MarkMpn/Sql4Cds
NuGet MarkMpn.SQL4CDS
Official site markcarrington.dev/sql-4-cds

What it does

Lets you query and manipulate Dataverse / D365 data and metadata with standard SQL syntax, including SELECT / INSERT / UPDATE / DELETE. When possible, queries are translated into FetchXML — so you can use familiar SQL to produce FetchXML, then use it in plugins or integrations.

Why it fits NL2SQL / TDS scenarios

  • The core engine is a standalone NuGet package and can be embedded directly into code or a pipeline without going through a GUI — ideal as a programmatic component in an eval pipeline.
  • It provides both .NET Framework 4.6.2+ and .NET Core 3.1+ assemblies.
  • The engine translates every SQL construct that has a direct FetchXML equivalent into FetchXML; for features with no equivalent (calculated fields, HAVING, etc.), it pushes the foldable filters into FetchXML as far as possible and computes the rest in memory.
  • Can go directly through the TDS endpoint: a connection property can tell it to use the preview TDS Endpoint for SELECT when feasible.
  • Includes a large set of SQL features not directly supported by FetchXML / TDS endpoint.

Advanced Capabilities

  • Metadata queries: converts SQL into RetrieveMetadataChangesRequest / RetrieveAllOptionSetsRequest.
  • CTE support (since v8.0.0): non-recursive CTEs expand into subqueries for TDS Endpoint compatibility; recursive CTEs convert to hierarchical FetchXML filters when possible.
  • JSON: OPENJSON, JSON_QUERY, ISJSON.
  • Performance: RetrieveTotalRecordCountRequest speeds up SELECT count(*), with automatic paging and workarounds for AggregateQueryRecordLimit.

Generated FetchXML includes the xmlns:generator="MarkMpn.SQL4CDS" namespace and tries to keep the full filter inline, for example:

<fetch xmlns:generator="MarkMpn.SQL4CDS">
  <entity name="contact">
    <attribute name="firstname" />
    <attribute name="lastname" />
    <attribute name="contactid" />
    <filter>
      <condition attribute="statecode" operator="eq" value="0" />
    </filter>
  </entity>
</fetch>

Availability

  • XrmToolBox plugin: xrmtoolbox.com/plugins/MarkMpn.SQL4CDS
  • SSMS / Azure Data Studio extensions
  • NuGet for programmatic integration: MarkMpn.SQL4CDS
Some queries cannot be executed with pure FetchXML. Those cases surface warnings indicating that SQL 4 CDS needs additional in-memory processing. This “fully foldable or not” signal is useful as an eval metric.

02SQL2FetchXML

Online ServiceKingswaySoftNo Install
URL sql2fetchxml.com
Help sql2fetchxml.com/Help.aspx

Paste SQL, click convert, and get FetchXML. It is good for a quick sanity check. It supports SELECT FROM WHERE AND OR LIKE GROUP BY ORDER BY AS TOP DISTINCT NOLOCK INNER JOIN LEFT OUTER JOIN ON IN plus aggregates COUNT/MAX/MIN/AVG/SUM, and automatically removes the Filtered prefix.

Operator notation

FetchXML-specific operators are represented as SQL functions with dashes removed: next-x-weeks → nextxweeks(n), today → today(), eq-userid → equserid(). If the operator itself is a SQL keyword (such as on or in), add an x prefix:

SELECT * FROM contact
WHERE createdon = xon('2011-01-11T00:00:00')
Known limitations: weak handling of parentheses and nested selects; occasional casing issues; cannot handle NOT IN / NOT EXISTS / EXCEPT (mostly FetchXML limitations anyway). For rigorous evaluation, use SQL 4 CDS as the main reference; use this tool only for quick comparison, not as ground truth.

03FetchXmlToSql Custom API

Official built-inUndocumentedReverse

The direction is opposite to your need (FetchXML → SQL), but it is an official path. Use Dataverse’s built-in but undocumented Custom API Function FetchXMLToSql:

GET https://{your_environment}.api.crm.dynamics.com
    /api/data/v9.2/FetchXMLToSQL(FetchXml=@p1)
    ?@p1='{Your_FetchXML}'

You can also call it from XrmToolBox with Custom API Tester. A common use case is converting FetchXML back to SQL when a virtual table points to SQL Server.

⚠️ Because this is undocumented functionality, there is no guarantee it will remain available. Do not depend on it in production products.

04FetchXML Builder

Open SourceXrmToolBoxMulti-syntax output
Plugin xrmtoolbox.com/plugins/Cinteros.Xrm.FetchXmlBuilder
Official site fetchxmlbuilder.com

Builds and executes FetchXML for Dataverse / Power Apps / D365. Its value for this topic is conversion across several syntaxes: it generates C# QueryExpression, WebAPI / OData query string, Power Automate List Records, SQL / JavaScript / C# stub. It pairs well with SQL 4 CDS — write SQL in SQL 4 CDS → edit it in Builder, and the reverse also works.

05FetchXML Studio

Open SourcePower Platform ToolBoxVisual
GitHub github.com/mohsinonxrm/pptb-mxrm-fetchxml-studio

Open-source visual FetchXML builder in the Power Platform ToolBox ecosystem. Its primary job is authoring FetchXML, but it integrates the undocumented FetchXMLToSQL API mentioned in §03 and provides SQL ↔ FetchXML bidirectional preview. Useful as another quick syntax cross-check tool.

06Official SDK sample: FetchXML ↔ QueryExpression

Open-source samplePowerApps-Samples

Docs: learn.microsoft.com — Convert queries sample

Demonstrates bidirectional conversion with QueryExpressionToFetchXmlRequest and FetchXmlToQueryExpressionRequest. It is not T-SQL→FetchXML, but if your pipeline already uses QueryExpression as an intermediate representation, this is the officially supported conversion path.

→Recommendations for Dataverse NL2SQL / TDS

  1. Programmatic Integration: Use the MarkMpn.SQL4CDS engine NuGet directly, feed model-generated T-SQL into it to convert to FetchXML, and compare whether the query is “fully foldable” as an eval metric.
  2. Pushdown capability is a good signal: If a query can be fully translated to FetchXML, it stays within Dataverse-native capabilities; parts that require in-memory processing can be tagged as “beyond FetchXML / possibly beyond the TDS endpoint.”
  3. TDS endpoint alignment: SQL 4 CDS exposes a connection property to toggle whether to use the TDS endpoint, which makes it easy to compare behavior between the FetchXML path and TDS path (matching your earlier work on OFFSET / EXCEPT / CTE support).
  4. Quick Online Lookup: Use SQL2FetchXML when you just want to see what a piece of SQL would look like as FetchXML, but do not treat it as ground truth.
