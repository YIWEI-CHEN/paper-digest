SECTION 01

Problem — Fine-tuned models "ace the test but can't do the job"

Text-to-SQL turns natural-language questions into executable SQL, letting non-experts query databases. In the LLM era there are two routes — prompting and fine-tuning — but both have hard limits in the real world. OmniSQL argues that fine-tuning open-source models on large-scale, diverse, high-quality synthetic data is the way to solve cost, privacy, and generalization at once.

The two dominant paradigms, and their pain

Prompting-based

Carefully crafted prompts on powerful closed-source LLMs (GPT-4 etc.). Downsides: high usage cost, data-privacy concerns, and little control over model behavior — all gated behind APIs.

Fine-tuning-based

Fine-tune on existing sets like Spider / BIRD. Downside: public data covers too narrow a slice, so models collapse when generalizing to complex or domain-specific databases.

The key observation (evidence of collapse): Qwen2.5-Coder-7B-Instruct fine-tuned on Spider + BIRD does well on same-distribution dev sets, but on out-of-domain ScienceBenchmark / EHRSQL it drops to just 43.8% / 31.4% execution accuracy — while zero-shot GPT-4-Turbo scores 59.2% / 43.1%. Fine-tuning overfits to the benchmark distribution.

Why not just augment existing data?

Prior data-augmentation work mostly grows new samples that hug the distribution of existing datasets, so diversity, quality, and scalability all suffer; many also need hand-built templates or grammars, which doesn't scale. OmniSQL wants a framework that is simultaneously:

Automatic

Minimal-to-no human intervention; the whole flow is LLM-driven.

Scalable

Generates large-scale, cross-domain, diverse data.

Realistic

Synthetic data aligns with real users' needs and scenarios.

Three contributions: (1) an automatic, scalable text-to-SQL data-synthesis framework; (2) SynSQL-2.5M — the first million-scale (2.54M-sample) text-to-SQL dataset, every sample carrying a CoT solution; (3) OmniSQL (7B/14B/32B), which sets a new open-source SOTA with far fewer parameters and matches top closed-source models.

SECTION 02

Framework — break the hard problem into four easy steps

The core tension is automation + scalability vs. quality + diversity. OmniSQL's answer is a progressive pipeline that decouples synthesis into four independent, individually controllable sub-tasks — each handled by a relatively small open-source LLM.

Fig 1 · OmniSQL's four-step progressive synthesis pipeline (redrawn from paper Figure 1)

Crucially the pipeline runs SQL → Question (synthesize the SQL first, then back-translate to natural language) — not the reverse. Why: translating SQL into natural language is "more accurate and less ambiguous" than going the other way, which guarantees the quality of each <question, SQL> pair.

To avoid a single LLM's stylistic bias, the paper uses multiple open-source model families across steps: Llama3.1 (8B/70B), DeepSeek-Coder (6.7B/33B, V2-Lite), Qwen2.5 (7B–72B), and Qwen2.5-Coder (7B–32B). Larger models are assigned more of the synthesis workload to keep quality high.

SECTION 03

Step 1 — Grow realistic databases from web tables

Real enterprise databases are scarce online because they hold sensitive data — but structured web tables are everywhere. OmniSQL's insight: hand the LLM a web table, ask it to infer a plausible business scenario behind it, then design a relational database that could store that data.

Two stages: generate + enhance

Table-driven database generation

Given a web table, prompt the LLM to first imagine the business scenario behind it, then design a database of several related tables. Each table carries a name, description, column names, column types, column descriptions, a primary key, and two sample rows; tables are linked by primary/foreign keys. The number of tables K is drawn from a normal distribution K ~ N(10, 4²) to encourage more complex schemas.

Database enhancement

First-pass schemas have two flaws: tables that are too simple (avg. 4 columns) and incomplete primary/foreign keys — a symptom of LLM "shortcut learning". A second enhancement step asks the LLM to add relevant columns to each table and complete missing primary/foreign-key relationships.

Enhancement, quantified: initial databases average 9.15 tables / 4.86 columns; after enhancement they reach 10.15 tables / 7.3 columns — real-world complexity. All databases are hosted in SQLite.

Why trust LLMs to write good schemas? The paper analyzed StarCoder's pre-training corpus and found 975,420 .sql files, 42% of which contain at least one CREATE TABLE — LLMs saw schema definitions extensively during pre-training. By contrast, GitSchemas / SchemaPile scrape DDL from GitHub (limited by the scarcity of high-quality SQL files), and WikiDBs stitches Wikidata tables together (risking joining unrelated tables and breaking consistency).

The actual prompt (excerpt from the repo)

The database-synthesis prompt asks the LLM to do three things at once: identify the domain, propose an enterprise-level scenario, and output a JSON schema — explicitly requesting "ultra-wide tables" of 15–50 columns. Excerpt from data_synthesis/database_synthesis/prompt_templates/schema_prompt.txt:

**Task Overview:**
Given a table in Markdown format, your task is to complete the following three tasks:

## Task 1
Analyze the table's header and rows to briefly summarize its domain.
The output domain should be surrounded by [START_DOMAIN] and [END_DOMAIN].

## Task 2
Based on the table, propose a complex, enterprise-level, real-world
business scenario ... surrounded by [START_SCENARIO] and [END_SCENARIO].

## Task 3

SECTION 04

Step 2 — Complexity-aware SQL generation

Asking an LLM to just "write SQL" yields a complexity imbalance: small models write trivial queries, large ones write overly complex ones. OmniSQL fixes this by defining four explicit complexity levels and, for each query, sampling one level and steering the LLM to match it.

Four complexity levels (per the repo's criteria)

Simple

Single table only, no joins allowed.

Moderate

Joins (INNER / LEFT / CROSS…) and richer WHERE clauses (IN / BETWEEN / LIKE).

Complex

Nested subqueries, self-joins, window functions, CTEs, complex WHERE/HAVING.

Highly Complex

Multiple CTEs, recursive CTEs, advanced window / analytical functions combined.

Beyond complexity, the prompt injects a few key components to keep queries meaningful and realistic:

  • Advanced SQL Functions: randomly sample a few engine-supported advanced functions (with names + descriptions), encouraged but not mandatory. This study uses the SQLite engine.
  • Database Values: randomly sample real values from a few columns to help the LLM write meaningful predicates.
  • Column Selection Constraint: constrain the number of columns the final SELECT returns, drawn from a geometric distribution (p = 0.6) — naturally biased toward small numbers, mirroring how real questions usually want just a few columns.

Post-processing: filter out the noise

After generation: (1) rule-filter out non-SELECT queries and actually execute the rest on the synthetic databases, dropping syntax errors and timeouts; (2) extract SQL templates (masking only values) and keep just one query per template for diversity. e.g. ... WHERE age > 18 and ... WHERE age > 55 share a template, so only one is kept.

The SQL-synthesis prompt itself is terse (excerpt from sql_synthesis_prompt.txt):

**Task Overview**
Create an executable SQL query based on the provided information.

**Database Schema**  {schema_str}
{sql_function_prompt}
{db_value_prompt}

**SQL Query Complexity**
Ensure the SQL query matches the {complexity} level ... {criterion}

**SQL Query Requirements**
1. Use the syntax specific to the {db_engine} database engine.
2. Incorporate advanced functions if appropriate, but ... not mandatory.
3. Address real-world data analysis needs. Avoid trivial ... queries.
4. (Very important) Ensure the final SQL query selects {column_count} columns.

**Answer**  Let's proceed step by step.
SECTION 05

Step 3 — Back-translate into questions, in nine styles

When back-translating SQL into questions, OmniSQL targets both semantic accuracy and linguistic diversity. Real users phrase questions in wildly different ways — from formal-and-explicit to vague-and-metaphorical — and a model trained on one register breaks on synonym substitution or paraphrasing.

Nine language styles

Clear intent (6)

formal, colloquial, imperative, interrogative, descriptive, concise — intent is clear, only the tone varies.

Needs inference (2)

vague, metaphorical — ambiguous words or figurative language needing external knowledge; the LLM also synthesizes that knowledge.

Multi-turn (1)

conversational — simulates a multi-turn <User> / <Assistant> dialogue where the user clarifies intent step by step.

One SQL, four registers (from the repo's style definitions):

Formal: Find all students older than 18 years and return their home addresses.

Colloquial: Hey! Could you help me find all the students who are over 18? I'd love to know their names and where they live.

Vague: What are the names and addresses of those older students?
(External Knowledge: 'older students' refers to age ≥ 18.)

Metaphorical: Find the names and addresses of those who have reached adulthood.
(External Knowledge: 'reached adulthood' refers to age ≥ 18.)

Semantic Consistency Selector

Each SQL yields multiple candidate questions. To pick the most semantically accurate one, embed candidates with Sentence Transformers (all-mpnet-base-v2), compute each candidate's average cosine similarity to all others, and select the one closest to the semantic center. Intuition: correct questions cluster together; mistranslations are the outliers.

selected = argmaxi ( 1/(N-1) · Σj≠i cos( emb(qi), emb(qj) ) ) q: candidate question; emb: all-mpnet-base-v2 sentence vector. Highest average similarity = closest to the semantic center.
SECTION 06

Step 4 — CoT solutions (that fix bad SQL along the way)

To bridge the gap between question and SQL, OmniSQL generates a chain-of-thought solution for every <database, question, SQL> triplet, spelling out the steps from question to query. This boosts interpretability and provides a high-quality training signal.

A typical CoT: analyze what the question asks → identify the relevant tables / columns / filters → assemble joins, filters, aggregations, and groupings step by step → emit the final SQL.

An unexpected payoff — CoT self-corrects: the paper found CoT-generated SQL sometimes differs from the original, and is usually a better fit for the question. The original triplets occasionally have minor issues (unnecessary columns, wrong join paths); step-by-step reasoning lets the LLM catch and fix them during synthesis.

The key design: the CoT prompt frames the original SQL as a colleague's reference solution, explicitly says it "may or may not be correct," and tells the LLM not to mention it. This nudges the LLM to override a wrong reference. Excerpt from cot_synthesis_prompt_template.txt:

You are a senior data analyst specializing in SQL. Your task is to
translate a natural language question into an executable SQLite query,
providing a detailed reasoning trace.

You will also receive a reference solution from a colleague, which
may or may not be correct. ... you are asked not to mention the
reference solution in any form. The reference solution might include:
1. Unnecessary table and column selections.
2. Incorrect or excessive joins.
3. Misalignment with the question.
4. Opportunities for simplification.

[Database Schema]: {schema}
[Natural Language Question]: {question}
[Reference Solution]: ```sql {sql} ```
Provide your step-by-step text-to-SQL solution here.

Reliability: majority vote. Multiple candidate CoTs are generated per triplet, grouped by their SQL's execution result, and the final CoT is picked from the largest group. A separate check: on 5,000 cases where execution results differed before/after CoT, GPT-4o judged which SQL better matched the question — 4,699 / 5,000 (93.98%) preferred the post-CoT version, confirming this step substantially improves data quality.

Quality: it beats human-annotated BIRD

GPT-4o as judge

Human experts (1,000 samples)

Three senior CS grad students gave pass/fail: 96% of databases realistic & complete, 97% of questions meaningful, 89% of SQL appropriate, 86% of full samples fully correct — high-quality "silver-standard" data.

SECTION 08

OmniSQL — model & training

Using SynSQL-2.5M plus the human-labeled Spider / BIRD data, the authors fine-tune OmniSQL at 7B / 14B / 32B, all built on the Qwen2.5-Coder-Instruct family.

How input / output is built

Input = database schema (formatted as CREATE TABLE statements) + the natural-language question. Each column is enriched via comments with three signals: column descriptions, representative values (two per column), and question-relevant values. Output = the CoT solution (step-by-step reasoning + final SQL). Since Spider / BIRD only ship gold SQL without CoT, the authors synthesize CoT for them using the Step-4 technique.

Loss = − E(x,y)∼D [ Σt log Pθ( yt | x, y<t ) ] Standard conditional next-token prediction: x = input (schema+question), y = output (CoT solution).

Training details

  • Base: Qwen2.5-Coder-7B / 14B / 32B-Instruct (pre-trained on 92 programming languages).
  • Fine-tuning: 7B and 14B use full fine-tuning; 32B uses LoRA (r = 256, α = 512, on q/k/v_proj) due to compute limits. LoRA versions of 7B/14B are also provided for comparison.
  • Hyperparameters: AdamW, batch size 512, 2 epochs, context length 8192; peak LR 2e-5 / 4e-6 / 2e-4; 5% linear warmup then cosine decay to 10%. bfloat16 + DeepSpeed-Zero.
  • Compute: one node of 8×A800-80GB; ~6 / 12 / 20 days for 7B / 14B / 32B. Inference via vLLM.

Standard benchmarks

OmniSQL-7B hits 87.9% on Spider test (greedy), beating the best public leaderboard method DAIL-SQL+GPT-4+SC (86.6%) by 1.3%. With majority voting the three scales reach 88.9 / 88.3 / 89.8%. On BIRD dev, OmniSQL-32B reaches 67.0% (Maj), competitive with fine-tuned-GPT-4o Distillery (67.2%).

Domain generalization

On the hardest, Spider2.0-SQLite: 7B baselines score only 0.7–3.7%, but OmniSQL-7B gets 10.4%. On EHRSQL (medical), OmniSQL-32B scores 46.8% (Maj), beating GPT-4o (45.5%).

Robustness

On Spider-Syn / Spider-Realistic (synonym substitution) it beats baselines in most cases. The one exception is Spider-DK (implicit domain knowledge), where 14B/32B trail their bases — see Limitations.

Tellingly, some open models (Qwen2.5-Coder-32B, Llama-3.1-70B) beat closed models on standard sets but lose that edge on domain-specific ones — likely because they ingested Spider / BIRD training data during pre-/fine-tuning. OmniSQL stays consistently strong across both.

Fine-tuning on SynSQL-2.5M alone gives +9.0% on BIRD and +12.9% on EHRSQL over the base. Removing SynSQL (keeping only CoT-enhanced Spider+BIRD) drops 7 datasets consistently — it fills the coverage gaps left by human data.

Data scale

SECTION 11

Limitations & takeaways

The paper is candid about a few boundaries and points to where the framework can extend.

Limitation 1 — implicit domain knowledge (Spider-DK): OmniSQL-14B/32B actually trail their base models on Spider-DK, which needs commonsense / implicit knowledge that the broadly-pretrained base has a stronger foundation in. Takeaway: add a new language style that injects commonsense knowledge.

Limitation 2 — single-step, single-model only: OmniSQL deliberately isolates the core "single LLM, single-step" capability and does not directly compare to multi-stage, multi-agent frameworks (CHASE-SQL, Alpha-SQL). There is headroom by adding schema linking, SQL revision, SQL selection, etc.

Limitation 3 — SQLite as the default dialect: the framework can in principle generate PostgreSQL / MySQL / BigQuery and other mainstream dialects (LLMs have seen them in pre-training), but this study uses SQLite to align with classic benchmarks. Cross-dialect validation is still pending.

Positioning: silver-, not gold-standard. Human eval shows 86% of full samples are fully correct — high-quality "silver-standard" data, good for training but not zero-error. The signal from scale outweighs the residual noise.

Three design tricks worth remembering

Go SQL→Question

Back-translation is more accurate and less ambiguous than forward translation — it locks in pair quality at the source.

Frame original SQL as a "reference"

Saying "may be wrong, don't mention it" lets the CoT override and fix it — 93.98% of cases get better.

Decouple → use small models

Four simple sub-tasks each go to a small open-source model, removing dependence on expensive closed LLMs — that's what makes million-scale feasible.

The dataset, all three model scales, and the full synthesis / training / evaluation code are open-sourced (SynSQL-2.5M under Apache 2.0). For anyone building their own text-to-SQL training data, this four-step pipeline is a directly copyable recipe.
