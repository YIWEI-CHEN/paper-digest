SECTION 01

The Problem — Semantic Drift

Enterprise data warehouses start with complete metadata: schema definitions, column semantics, business context. The moment data flows through multi-stage, multi-language pipelines, that metadata goes stale.

The paper names this semantic drift (citing Müller & Stein 2016). In practice it looks like this: nobody can say precisely which raw columns, transformations, and aggregations produce the MAU, churn rate, or revenue number on a dashboard. That knowledge lives in the heads of the few engineers who wrote the pipeline, or in stale hand-maintained docs.

Why this matters more in the LLM era: semantic drift directly damages two downstream systems — RAG (if the schema descriptions in the knowledge base are wrong, the retrieved context is wrong) and text-to-SQL (a model that doesn't know what a column actually means produces syntactically valid, semantically wrong SQL). This framing is why the paper positions lineage extraction as AI infrastructure rather than a data-governance chore.

Why prior work falls short

  • Classical program analysis (AST parsing, metadata mapping, runtime analysis — sqlparse, sqlfluff, Purview): reliable in single-language, static settings, but breaks on multi-stage multi-language pipelines, and every codebase change means rewriting rules.
  • LLM-based extraction (Li et al. 2025; Cyberhaven's LLiM): reframing lineage extraction as code understanding does work, but existing CoT approaches generate table-level and operation-level lineage separately, missing the composed semantics. Fine-tuned approaches like LLiM depend on enterprise-internal positive/negative event labels and generalize poorly.

This paper's stance is a no-fine-tuning paradigm: emit table-level and operation-level lineage in a single query, on a dataset curated for data understanding rather than anomaly detection.

Four contributions

01 Formalization

A formal schema-lineage definition for multi-language enterprise pipelines, covering source-to-output semantics, transformation, and aggregation.

02 Benchmark

50 real production scripts, 1,700 manually annotated lineages.

03 SLiCE Metric

A decomposable composite evaluation framework spanning structure and semantics.

04 Experiments

12 language models × 7 prompting strategies × 6 random seeds.

SECTION 02

Schema Lineage as a Four-Key Dictionary

The paper defines schema lineage as a structured dictionary with exactly four required keys. The "exactly four" is not just notation — it doubles as the evaluation interface and the downstream parsing contract.

L = {
  source_schema  : C  // set of source columns
  source_table   : T  // set of source tables
  transformation : F  // transformation logic (code)
  aggregation    : A  // aggregation operations
}

Task setup: given a pipeline script S (potentially a mix of SQL, Python, PySpark, and C#) and a target schema σ in the output table, extract the corresponding lineage L.

Source Schemas (C)

The original columns the lineage originates from. Multiple sources are comma-separated.

Source Tables (T)

The initial tables containing those columns — the data's root origins.

Transformation (F)

The code snippet converting source columns into the target schema. Sequences delimited by <CODEEND>.

Aggregation (A)

GROUP BY / SUM / COUNT / MAX / MIN plus grouping keys, same <CODEEND> delimiter.

Fig 1 · Schema lineage annotation (redrawn from paper Figure 1)

Design point: lineage is traced for a single column, not a table. That's why 50 scripts yield 1,700 annotations — each script's output table has many columns, each with its own independent lineage (34 on average per script).

SECTION 03

Dataset — 50 Real Scripts, 1,700 Annotations

The hardest-to-replicate and most valuable asset in the paper: all scripts are actively deployed inside Microsoft, spanning SQL, C#, and Python (including PySpark), serving business metrics, marketing analytics, product insights, and user-experience work.

Difficulty scoring (Appendix A.1)

Scripts are scored 0–3 across three independent dimensions, +1 point each:

Data Sources

≥ 3 distinct data sources → +1

Transformation

A transformation chain (one operation's output feeds another) → +1

Aggregation

Any aggregation function (SUM/COUNT/PIVOT) → +1

Total 0–1 = Easy, 2 = Medium, 3 = Hard.

SECTION 04

SLiCE — Schema Lineage Composite Evaluation

The methodological core. SLiCE compresses structural validity and semantic correctness into a single [0,1] score while keeping five component-level diagnostics (format, source schema, tables, transformation, aggregation).

The master equation: two binary gates × a weighted sum

SLiCE(L̂, L*) = MFMT(L̂) · MSRC(L̂,L*) · [ ωTBL·MTBL + ωTRF·MTRF + ωAGG·MAGG ]
ω_TBL + ω_TRF + ω_AGG = 1　·　paper setting: ω_TBL = 0.4, ω_TRF = 0.4, ω_AGG = 0.2

The single most important property: the first two terms are multiplicative gates, not weighted addends. MFMT and MSRC are both binary (0 or 1). If the output format breaks, or the source-schema set is not matched exactly, the entire lineage scores zero regardless of how good the remaining three components are. The paper's justification is that foundational structural errors propagate downstream in real systems. This design is why DeepSeek-Coder-1.3B scores exactly 0.000 under base prompting in Table 2.

MFMT — format correctness (binary)

Two accepted formats, depending on prompting strategy:

# with reasoning trace
<think> ... reasoning trace ... </think>
<answer> ... schema lineage dictionary ... </answer>

# without reasoning trace
<answer> ... schema lineage dictionary ... </answer>

The content inside <answer> must be a strict key-value dictionary with exactly four keys. Any malformed tag, wrong key name, or missing field → immediate 0.

MSRC — source schema (binary, exact set equality)

MSRC(L̂,L*) = 1  if  Ĉ = C*　　otherwise 0
case-sensitive, order-insensitive set matching

MTBL — source tables (exact F1 + fuzzy hybrid)

Because naming conventions vary (database.schema.table vs bare table), pure exact match is too strict:

MTBL = w1TBL · F1(T̂,T*) + w2TBL · Fu(T̂,T*)
paper setting: w₁ = 0.7 (exact F1), w₂ = 0.3 (fuzzy)

Fu(T̂,T*) = ½ [ (1/|T̂|)·Σ max FuzzyMatch(ti,tj) + (1/|T*|)·Σ max FuzzyMatch(ti,tj) ]
FuzzyMatch = normalized Levenshtein similarity; first term is fuzzy precision, second is fuzzy recall

MTRF / MAGG — transformation & aggregation (CodeBLEU variant + Multi-AST)

The difficulty here is logically equivalent but syntactically different code. The paper introduces Multi-AST similarity for multilingual code comparison:

ASTmulti(x̂,x*) = Σl∈L wl · ASTl(x̂,x*)
w_l = confidence that x belongs to language l, estimated as the normalized proportion of language-specific keywords; Σ w_l = 1

MTRF = w1·BLEU + w2·BLEUweight + w3·ASTmulti
MAGG = w1·BLEU + w2·BLEUweight + w3·ASTmulti
paper setting (same for TRF and AGG): w₁ = 0.5, w₂ = 0.3, w₃ = 0.2

How this differs from CodeBLEU (a likely interview question): SLiCE keeps CodeBLEU's three foundations — n-gram BLEU, weighted BLEU, AST similarity — with two deliberate changes: (1) single-language AST similarity is extended into a language-aware weighted aggregation to handle snippets mixing several languages; (2) the data-flow matching term is deliberately removed, because lineage transformations are frequently partial and non-executable code fragments that no data-flow analysis can process. For the same reason, execution-based metrics like pass@k are inapplicable here.

Evaluation protocol: schema-level → script-level → corpus-level

MSCR(si,Θ) = (1/Ki) · Σk=1..Ki SLiCE(L̂ik, L*ik)　// average over K_i schemas in a script
MMOD(Θ) = (1/I) · Σi=1..I MSCR(si,Θ)　// average across 50 scripts

Note this is a macro-average — within-script first, then across scripts. Every script carries equal weight, so hard scripts with many columns do not dominate the headline number.

SECTION 05

Prompting — Three Categories, Seven Variants

Three hierarchical prompting categories, designed to systematically test how contextual richness affects extraction quality.

Base Prompting (zero-shot)

Only the pipeline script plus explicit extraction instructions (target output format and component definitions). The "minimum necessary context" baseline.

Few-Shot Prompting (one / two / three-shot)

Base plus concrete input-output example pairs. Example count scales with script complexity: one for easy, up to two for medium, up to three for hard.

Chain-of-Thought (CoT-1 / CoT-2 / CoT-3)

Few-shot, but the examples are replaced with detailed human-written reasoning traces demonstrating step-by-step derivation of lineage from pipeline code.

The three categories expand into 7 concrete strategies: base, one-shot, two-shot, three-shot, CoT-1, CoT-2, CoT-3.

An engineering detail worth raising (shows systems depth): open-source SLMs are served with PagedAttention (vLLM's KV-cache virtualization). The key observation is that within a single script, the prompt prefix is identical across all target schemas — the script, the extraction instructions, and the examples never change; only the target column name does. So KV pairs are computed and cached once per pipeline rather than once per schema. At 34 schemas per script on average — and up to 391 for the hardest — that eliminates an enormous amount of redundant computation.

SECTION 06

Results

18 models were evaluated initially (2 LLMs + 16 SLMs). Six were dropped for excessive inference time or consistently poor performance, leaving 12 in the final results.

Model selection

Kept (12): GPT-4.1, GPT-4o; Qwen2.5-Coder (1.5B / 3B / 7B / 14B / 32B), Mistral-7B, Codestral-22B, DeepSeek-Coder (1.3B / 6.7B), Phi-4 (14B).

Dropped (6): CodeLlama (7B / 13B / 34B), Phi-4-mini, Phi-4-reasoning-14B, and one further variant.

Scale: 12 models × 50 scripts × 7 strategies × 6 random seeds = 50,000+ extraction tasks.

Four key findings

① The first example buys the most. Base → one-shot lifts GPT-4.1 by 61% and Qwen2.5-Coder-32B by 75%. General-purpose models (GPT-4o, Mistral, Phi-4) gain over 100% — because their base scores were locked down by the format gate (Phi-4 goes from 0.016 to 0.511). Read carefully, this says: one-shot's primary job is teaching output format, not teaching lineage extraction.

② CoT actively hurts models below ~3B. DeepSeek-Coder-1.3B (0.054 → 0.038) and Qwen2.5-Coder-1.5B (0.309 → 0.304) both regress moving from one-shot to CoT-1. The paper offers two explanations: CoT reasoning is an emergent capability of larger models (Wei et al.), and longer CoT prompts overwhelm small context windows (echoing Liu et al., "Lost in the Middle"). Mistral-7B shows the same regression (0.331 → 0.227), suggesting this is not purely about parameter count but also about whether the model is code-pretrained.

③ A 32B open model ≈ the GPT series — the paper's headline. Qwen2.5-Coder-32B reaches 0.734 under CoT-1, against GPT-4o's 0.759 and GPT-4.1's 0.767. The sharper version: its base score (0.355) already beats GPT-4o's (0.284), and its CoT-1 performance exceeds GPT-4o's one-shot at every difficulty level. For enterprises this means internal deployment is viable — no need to ship pipeline scripts containing sensitive business logic to an external API.

④ One high-quality reasoning trace is enough; returns diminish fast. On hard scripts, Qwen-32B gains 23% from one-shot → two-shot (0.531 → 0.653) but only 6% from CoT-1 → CoT-2 (0.689 → 0.727). Phi-4 scores 0.660 on hard scripts with CoT-1 versus 0.397 with one-shot. Conclusion: CoT meaningfully offsets the complexity penalty, but marginal returns decay quickly.

The difficulty-stratified analysis (Figure 2a) confirms the intuition: all models degrade as script complexity rises; but moving from one-shot to CoT-1 lifts every model at every difficulty level, meaning CoT effectively mitigates the complexity penalty.

SECTION 07

Discussion, Limitations, Future Work

Limitations the authors state

Cost: proprietary LLMs perform well, but every prompt must carry the complete pipeline script, which can exceed hundreds of thousands of tokens, escalating cost.

Scalability bottleneck: CoT's effectiveness depends on human experts writing reasoning traces for each script type. This requirement limits scalability — and it is the paper's explicitly named principal weakness.

The authors' stated future direction (← the mandate for any extension)

The paper flags the same idea in two places, in language that reads like it was written to license follow-up work:

§3.2: "the structured formulation of the SLiCE metric is not only useful for evaluation but also well-suited to serve as a reward signal in future supervised fine-tuning or reinforcement learning frameworks" (citing DeepSeek-R1 and execution-based RL code generation).

§7: "This metric … provides a well-structured reward signal that can be leveraged for fine-tuning language models in future work, offering a direct path toward improving model alignment with schema lineage extraction tasks."

In other words: this paper performs no fine-tuning at all — it is entirely prompting-only. Using SLiCE as an RL reward is the next step the authors themselves declared in the body text.

The downstream narrative

The paper uses TotalAmountSpent to illustrate how lineage feeds RAG: knowing it derives from customer_id and amount via a SUM aggregation lets an LLM generate a precise business statement ("TotalAmountSpent shows the total amount spent by each customer by aggregating individual transaction amounts"). For text-to-SQL, accurate lineage supplies precise column definitions and business context, directly improving generation quality.
