SECTION 01

Problem — Memory is a skill, not a module

Cognitive science calls the ability to decide "what is worth remembering, when to retrieve it, and how to organize what is known" metamemory — and it is a learned skill. This paper brings that view to LLMs: rather than designing memory as a fixed architectural module, let the model actively manage memory and treat that as a skill that can be observed, evaluated, and optimized.

1.1 The LLM bottleneck: the context window is working memory

An LLM's context window plays the role of working memory — a fixed-size buffer bounding what it can attend to at once. Long-horizon tasks routinely run tens of thousands of steps, far exceeding this. Prior external-memory designs (retrieval databases, vector stores, scratchpads, summary buffers — MemGPT, Generative Agents, A-MEM, MemoryBank) almost all treat memory as a fixed mechanism designed into the system.

1.2 A different view

AutoMem promotes file-system operations — read / write / search / append / create — to first-class memory actions in the model's action space, on equal footing with the actions it uses to act on the world. The same forward pass that picks a task action can also select a memory operation (e.g. <|APPEND|> or <|SEARCH|>). This minimal design gives the model full control over external memory while keeping behavior cleanly observable — every memory decision is a traceable action in the trajectory.

Two axes that resist manual optimization: a memory skill improves along (1) the structure that supports it (prompts, file schema, validation logic, action vocabulary), and (2) the model's proficiency at exercising it. But an episode can span 105 steps, and a mistake made at step 50 — failing to log a coordinate, or writing a duplicate that buries useful info — may not surface until step 800. Human review of full trajectories is impractical.

1.3 Key insight: let a meta-LLM be the "code reviewer"

The core observation: a sufficiently capable LLM (acting as a meta-LLM) can read an agent's complete episode (thousands of steps) and identify where memory decisions went wrong — much as a code reviewer reads a full execution log. This makes automating both axes of memory improvement feasible.

1.4 Contributions

① Memory as a learnable skill

Instantiated via file-system operations in the same action space as task actions, giving the model full, observable control over what to encode, when to retrieve, and how to organize.

② The AutoMem framework

Automates improvement along two complementary axes: scaffold revision (iterating the memory structure) + targeted training (sharpening memory proficiency). Both loops are driven by meta-LLMs analyzing complete traces.

③ Closing the frontier gap

Across three procedurally generated long-horizon games, targeting memory alone yields ~2×–4× progression gains, significantly closing the gap between a 32B open-weight model and frontier proprietary systems.

SECTION 02

Method Overview — one inner loop + two outer loops

AutoMem consists of two sequential outer loops operating on a shared inner-loop agent that uses a file system as its memory. Outer-loop #1 pushes the agent scaffold as far as code revision can go; outer-loop #2 then trains the model's memory ability beyond the ceiling of any fixed scaffold.

Fig. 1 · AutoMem overview (redrawn from paper Figure 3): two automated outer loops optimize a shared inner-loop agent

2.1 Inner-loop agent: memory as a file system

The inner loop is a single LLM agent executing one episode, equipped with a directory of files on disk as external memory. Each step runs two routines:

LOG — "what is worth recording about what just happened?"

The agent decides whether and how to record the environment's response to the previous action: append to an existing file, create a new one, or rewrite an entry.

PLAN — "what do I need to recall to act now?"

The agent searches across files, reads specific entries or their tails, and commits the next world action.

Why the shared action space matters: because memory operations and task actions live in the same trajectory as traceable actions, the outer loops can observe, evaluate, and optimize them. Experiments confirm: optimizing the memory structure also improves task behavior (e.g. gameplay) — better-organized memory reduces redundant exploration and directionless action, even though the optimizer targets the scaffold rather than the task strategy.

SECTION 03

Two loops, one objective

Both loops mirror the familiar structure of machine learning: each has parameters θ to optimize and an update signal ∇L, both derived from a meta-LLM's trajectory analysis. The scaffold sets the structural ceiling for what memory operations are possible; proficiency training pushes the model toward that ceiling.

Loop #1: θ = agent scaffold (code / prompt / schema)　　∇L = a code revision
Loop #2: θ = dedicated memory-model weights　　∇L = a properly-configured supervised training step
Shared principle: long-horizon improvement = trajectory-level review + targeted revision

3.1 Outer-loop #1: optimizing the memory scaffold (structure)

The optimization signal must be trajectory-level because memory consequences are delayed in long-horizon tasks: failing to log a coordinate or writing a duplicate at step 50 may not surface until step 800, when the agent gets lost or re-explores. Final-return metrics alone discard the structure that reveals where memory went wrong.

So the meta-LLM is given the full episode trace (per-step logs, the resulting memory directories, and the agent code itself) — functioning as a code reviewer with a complete execution log, not a scalar reward. Reviewing NetHack traces, it found an unbounded map file accumulating thousands of duplicate coordinate entries and introduced a clean coordinate-keyed deduplication format. Each iteration is gated on measured improvement: the rewritten agent replays the same fixed seeds, and a revision is kept only if average progression improves — converging in ~2–5 iterations.

3.2 Outer-loop #2: training memory proficiency

Once the scaffold is optimized, the remaining gap is the model's parametric ability to make good memory decisions. This loop uses the meta-LLM as a training engine to update weights. It reads the agent code plus a pool of episode traces, derives selection criteria, and produces supervised training data.

The meta-LLM is a filter, not a teacher: every training example is verbatim text the inner-loop model itself produced during an episode; the meta-LLM only selects which responses to reinforce — filtering the model's own behavior rather than generating new responses. It also orchestrates data-selection logic, data composition, and the LoRA training configuration as a joint decision, refined across iterative trials so finetuning delivers a clean lift rather than disruption.

3.3 Memory specialist vs gameplay model: a clean split

Because memory is a separable skill, the paper finetunes only a dedicated memory model (memory specialist) while the model that commits world actions stays unmodified. At inference the inner loop runs two model instances sharing one conversation history:

Memory specialist (LoRA-finetuned)

Handles the LOG routine and the memory-consultation part of PLAN. It is a LoRA-finetuned copy of the base model.

Gameplay model (frozen)

The unmodified base model, which commits the world action. After the specialist's last memory operation, a short handoff passes the conversation to it.

Two benefits: (1) the training signal stays focused — the supervised loss targets memory-operation behavior without being diluted by action-format examples; (2) the base model's competence at well-formatted world actions is fully preserved, since gameplay weights are never touched. As a result the memory-proficiency gain stacks cleanly on top of the scaffold gain (a further ~9–18% relative gain) rather than trading off against it.

SECTION 04

Environments & Metric — why procedurally generated games

Procedurally generated long-horizon games are well suited to studying memory: episodes are long enough that context-window management alone can't sustain performance; worlds regenerate each episode so pretraining knowledge transfers poorly; and success demands the records humans naturally keep — maps, inventories, encounter logs, strategy notes. All three come from the BALROG benchmark.

Crafter

2D open-world survival. 17 actions, 22 achievements (exploration/crafting/combat). Episodes up to ~103 steps.

4.1 Metric & setup

Metric: progression rate (0–100)

Crafter = fraction of 22 achievements; MiniHack = fraction of 8 tasks completed (binary per task); NetHack = BALROG's dungeon-and-experience-level progression. Reported as mean ± standard error over 10 fixed seeds [42..51].

Model & meta-LLM

Inner-loop base = Qwen2.5-32B-Instruct (served locally with vLLM). Scaffold optimizer uses Claude Opus 4.6, training engine uses Claude Opus 4.7 as the meta-LLM. The memory specialist is LoRA-trained; the gameplay model is the unmodified base.

Baselines fall in two groups: (i) BALROG leaderboard — frontier proprietary (Gemini-3-Pro, Gemini-3.1-Pro-Thinking, Claude-Opus-4.5, Gemini-2.5-Pro), frontier open-weight reasoning (DeepSeek-R1, 671B), and same-family open weights at two scales (72B, 7B); (ii) the same 32B under the BALROG harness with basic context management: a 16-step sliding window, with/without chain-of-thought.

5.1 Memory is a high-leverage axis on its own

Scaffold optimization (weights untouched) doubles or more than triples progression: Crafter 25.0→47.27 (×1.89), MiniHack 7.5→27.5 (×3.67), NetHack 0.42→1.57 (×3.74). The scaffolded 32B agent beats Qwen2.5-72B-Instruct by a wide margin and the same base under sliding-window by an even wider one — indicating that on long-horizon tasks, well-structured external memory management is higher-leverage than model scale or context management.

Fig. 2 · Crafter progression across the three stages (values from paper Table 1 / Figure 1): v0 → scaffold → +train, matching Claude-Opus-4.5

5.2 Training the specialist adds complementary lift on top of the scaffold

On top of the optimized scaffold (Crafter v5 / MiniHack v4 / NetHack v2), memory-proficiency training lifts progression to Crafter 51.36% (+4.09), MiniHack 30.0% (+2.5), NetHack 1.85% (+0.28) — comparable to one or two scaffold iterations. The two axes work in sync: structure revision raises the prompt/schema ceiling, proficiency training addresses the remaining capacity gap — together bringing the 32B model to Claude-Opus-4.5's level (49.5/27.5/2.0) and within a few points of Gemini-3.1-Pro-Thinking (55.0/27.5/2.6).

5.4 Memory file schema evolution (NetHack example)

Below is a concrete schema revision outer-loop #1 produced for NetHack (from paper Figure 5): it replaces an unbounded append-only map file with a coordinate-keyed deduplication format, auto-syncs inventory/status, and pre-populates a strategy reference. This shrinks per-step memory growth from 138 characters to 6 — a 95% reduction.

# v0: unbounded, append-only, accumulates duplicate coordinates that bury useful info
dungeon_map.txt:
  (3,4) corridor
  (3,4) corridor      # duplicate: same tile logged again on revisit
  (3,5) wall
  (3,4) corridor      # duplicate again... file grows without bound

# Optimized: coordinate-keyed dedup + auto-synced files + strategy reference
<|UPSERT_MAP|> (3,4) corridor   # new observation overwrites the old entry, no append
inventory.txt   # auto-synced by the scaffold from observation; no manual READ/reconcile
status.txt      # likewise maintained programmatically
strategy.txt    # pre-loaded primary objective (find stairs, descend); no rediscovering the goal

5.5 Training internalizes a "consult-before-write" discipline

The trained memory specialist shows a consistent shift toward consulting memory before modifying it. In the LOG phase, the ratio of writes to SEARCHes falls in every environment — the specialist searches existing files before appending, rather than writing blindly.

SECTION 06

Limitations & Future Work

The authors explicitly call out three limitations, each pointing to an extension.

① Only episodic memory studied

The file system starts fresh each episode. A natural extension is persistent memory that carries knowledge across episodes.

② Only validated on games

Games are well-suited to studying memory (long horizons, procedural generation, rich information-management demands), but the approach could also apply to real-world, memory-intensive tasks.

③ One scaffold/specialist per environment

The three games differ in structure and objectives, so a separate scaffold and specialist are optimized for each; whether a single shared one works is unexplored.

Conclusion

This work shows memory management is an independently learnable, high-leverage skill for LLM agents. AutoMem factors it into structure and proficiency and automates both via meta-LLM outer loops; targeting memory alone (without touching gameplay weights) improved performance ~2×–4× across three long-horizon games, approaching frontier proprietary systems. A broader finding: long-horizon task improvement can decompose into trajectory-level review + targeted revision — a workflow meta-LLMs can execute autonomously where human review of full traces is intractable. Applying this decomposition to agent capabilities beyond memory is a promising direction.

Broader impacts: automating scaffold optimization and targeted training of memory management as a distinct capability lowers the model-scale threshold at which long-horizon agents become practical — an accessibility gain for open-weight deployment — and the techniques could extend beyond games. The authors caution that the released artifacts are not directly applicable to high-stakes deployment without further safety review.
