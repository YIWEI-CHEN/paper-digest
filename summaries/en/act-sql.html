ACT-SQL

Using "automatically generated chain-of-thought" for Text-to-SQL in-context learning — no manual annotation, one API call to generate SQL, and SOTA among in-context learning methods on the Spider dev set.

SECTION 01

Problem Statement

The goal of Text-to-SQL is to translate a natural-language question into the corresponding SQL query given a database schema, so ordinary users can access relational databases in a more intuitive way.

Prior work centered on cross-domain semantic parsers: in datasets such as Spider, SParC, and CoSQL, the databases used in train / dev / test do not overlap. The traditional approach trains dedicated text-to-SQL models (such as RAT-SQL, LGESQL, and PICARD). Although their scores are strong, they have three structural pain points:

Expensive data

They require large amounts of high-quality manually annotated training examples, making dataset construction costly.

Slow training

Finetuning consumes substantial compute resources and time.

Hard to deploy

The model structures are complex (for example, AST-based), making productionization and maintenance difficult.

The emergence of LLMs opens a new path through in-context learning: no finetuning is needed; zero-shot / few-shot prompts can already work. But the authors point out that LLM-based methods at the time still had clear gaps:

① Simple prompts perform poorly: prompt designs such as Rajkumar et al. (2022) are too plain and lag behind finetuned models.

② Complex pipelines are slow and expensive: DIN-SQL uses a multi-step workflow and needs to call the LLM API multiple times (around 4) to generate one SQL query.

③ No automatic CoT: although CoT improves reasoning, it requires manual example selection and annotation; text-to-SQL had no method for automatically generating CoT.

④ Not extended to multi-turn settings: in-context learning methods had not yet been expanded to multi-turn dialogue datasets such as SParC and CoSQL.

Three Core Contributions

Exploring the prompt-design space

The paper systematically studies how DB schema presentation formats and few-shot exemplar selection strategies affect LLM performance.

Proposing ACT-SQL automatic CoT

It automatically generates chain-of-thought rationales, requiring only one API call per SQL query, saving time and cost while achieving in-context-learning SOTA on the Spider dev set.

Extending to multi-turn Text-to-SQL

It reaches performance comparable to finetuned models on SParC and CoSQL.

SECTION 02

Methodology

The authors formalize SQL generation as a single LLM call. Performance depends on three variables: the database presentation format, the exemplar selection strategy, and the design of the exemplar prompt.

S = LLM( I , D , Q , E )
I = instruction  ·  D = database schema  ·  Q = question  ·  E = [(D₁,Q₁,P₁), …, (Dₙ,Qₙ,Pₙ)] exemplar list, where Pᵢ is an answer prompt containing the gold SQL

2.1 Database Presentation Format (DB Prompt Style)

Under zero-shot evaluation, the schema presentation format almost dominates LLM performance. The authors compare five formats:

Five DB schema presentation formats
FormatDescription
Table(Column)Lists one table and its columns per line, following the example in OpenAI's official documentation.
Table(Column)(PF)Adds primary-key (PK) and foreign-key (FK) information to the end of the previous format.
Create(NoPF)Uses SQL create table syntax to describe the schema, including column types but no PK/FK.
Create(EoC)Based on Create, adds PK/FK information at the end of the corresponding column (End of Column).
Create(EoT)Based on Create, adds PK/FK information at the end of the whole table (End of Table).

In addition, c example data rows (DB contents) are appended after each table to help the model infer actual stored values (for example, distinguishing "France" from "French").

create table singer (
  Singer_ID number,
  Name text,
  Country text,
  ...
  primary key (Singer_ID)
)

2.2 Exemplar Selection — Hybrid Strategy

The authors use a hybrid static + dynamic strategy. Each test case receives ns + nd exemplars:

Static exemplars ns

Randomly sample ns examples from the training set and use them in the context of every test case.

Dynamic exemplars nd

For the current test question, compute question similarity with a pretrained model and choose the top-nd most relevant examples from the train set.

Core intuition: the closer a dynamic exemplar's question is, the more useful information it can provide to the LLM.

SECTION 03

Core: ACT-SQL Automatic CoT

The authors design the CoT prompt in a form "similar to schema linking" — but the key is that it can be automatically generated, with no manual annotation at all.

Traditional schema linking maps and connects question tokens to schema items (tables / columns) through string matching. ACT-SQL follows the same spirit: for each training example (schema + question + gold SQL), it automatically derives a reasoning chain indicating which question slice corresponds to which column/table.

Automatic Generation Algorithm

Given a question q = (q₁,…,q|q|) and SQL s, define a slice qᵢ,ⱼ = (qᵢ,…,qⱼ). The pipeline has three steps:

Enumerate columns in the SQL

For each [tab].[col], use a pretrained model to compute its similarity to all question slices, and link it to the most relevant slice. Ignore columns in the GROUP BY clause (they are usually not mentioned directly in the question).

Enumerate tables in the SQL

Exclude tables that already appeared in step ①. For the remaining tables that only appear in FROM, likewise find and link the most relevant question slice.

Enumerate values

Add the values appearing in the SQL to the CoT, then append the complete SQL to the end of the CoT.

Most relevant slice  =  argmaxqᵢ,ⱼ Sim( [tab].[col] , qᵢ,ⱼ )
Figure 1 · ACT-SQL automatic CoT generation pipeline (redrawn from Figures 1–2 in the paper)

Example of Automatically Generated CoT (Paper Table 13)

Question: "Find the package choices and series names of TV channels that have high-definition TV." The few-shot version mistakenly places Hight_definition_TV into SELECT. After applying ACT-SQL, the model first performs complete schema linking and finally writes the correct SQL without redundant columns:

Let's think step by step.
According to "TV channel that has high definition TV",
  columns [TV_Channel.Hight_definition_TV] may be used.
According to "package choice and series name",
  columns [TV_Channel.Package_Option] and
  [TV_Channel.series_name] may be used.
So the final answer is:
SELECT package_option , series_name
FROM TV_Channel WHERE Hight_definition_TV = 'yes'

Key advantage: the whole reasoning chain is generated by the LLM in the same output — ACT-SQL does not need extra LLM API calls to produce CoT, so it is faster and cheaper than multi-step methods such as DIN-SQL.

SECTION 04

Extending to Multi-turn Text-to-SQL

Multi-turn questions depend on shared context, so single-turn prompts and auto-CoT cannot be applied directly (schema-linking information may be scattered across multiple utterances).

The authors adopt a two-stage method: first use an LLM to rewrite multi-turn questions into standalone questions with context dependencies removed, convert the multi-turn dataset into a single-turn dataset, and then apply the in-context learning method above. For each multi-turn dataset, they manually annotate 10 rewriting examples to fix the format and improve quality.

Known weakness: rewriting quality directly affects the score. If rewriting omits key information (for example, the word "airline" from the first sentence disappears in later turns), the error propagates through schema linking — this is also the main reason ACT-SQL performs worse on multi-turn datasets than on single-turn ones.

Evaluation Metrics

EM

Exact Match: every component of the predicted SQL must be equivalent to the gold SQL (values are not compared).

EX

Execution Accuracy: the execution result must be correct, usually more precise than EM.

TS

Test-Suite: the execution result must be correct across multiple database instances with the same schema.

For multi-turn evaluation, the paper additionally uses QM (question-level) and IM (score only if the entire interaction is correct).

SECTION 06

Experimental Results

The primary model is the low-cost GPT-3.5-turbo; GPT-4 is used on Spider to validate auto-CoT. The similarity model is text2vec-base-chinese; temperature = 0 (greedy).

SECTION 07

Prompt Source Code — From the GitHub Repo

Below are the key code snippets directly extracted from X-LANCE/text2sql-GPT, showing how ACT-SQL actually assembles prompts. The examples in Appendix C of the paper all come from these functions.

7.1 System message — just one sentence

Whether zero-shot or few-shot, the system message given to the Chat model is always this line (from util/prompt.py get_prompt()):

# util/prompt.py · line 66
prompt = [{'role': 'system',
           'content': 'Given the database schema, you need to '
                      'translate the question into the SQL query.'}]

Note that there is no complex role-playing, no rule list, and no few-shot instruction preamble. The whole system prompt has only 14 words; all "teaching" is conveyed through subsequent user/assistant exemplar dialogues.

7.2 Main prompt assembly function get_prompt()

This is the core entry point of the paper's method: it concatenates the system message, several examples of (user question + assistant SQL or CoT), and finally the actual test case's user question into one multi-turn dialogue.

# util/prompt.py · get_prompt() · simplified
def get_prompt(self, args, db_id=None, question=None, shots=[], c_num=-1):
    if c_num < 0:
        c_num = args.content  # number of DB example rows (paper uses 3)

    if args.gpt in GPT_CHAT_MODELS:  # gpt-3.5-turbo / gpt-4
        prompt = [{'role': 'system',
                   'content': 'Given the database schema, you need to '
                              'translate the question into the SQL query.'}]

        # ===== Append ns + nd examples one by one (static + dynamic) =====
        for shot in shots:
            prompt.append({'role': 'user',
                'content': f"Database schema:\n{self.db_prompts[shot['db_id']][c_num]}\n"
                           f"Question: {shot['question']}"})
            if args.cot:
                prompt.append({'role': 'assistant', 'content': shot['cot']})
            else:
                prompt.append({'role': 'assistant', 'content': shot['query']})

        # ===== Finally append the actual test case to be answered =====
        if db_id and question:
            prompt.append({'role': 'user',
                'content': f'Database schema:\n{self.db_prompts[db_id][c_num]}\n'
                           f'Question: {question}'})

    elif args.gpt in GPT_COMPLETION_MODELS:  # code-davinci-002, text-davinci-003
        prompt = ''
        for shot in shots:
            prompt += 'Given the database schema:\n'
            prompt += self.db_prompts[shot['db_id']][c_num] + '\n'
            prompt += 'Translate the natural utterance into the SQL query: ' \
                      + shot['question'] + '\n'
            prompt += (shot['cot'] if args.cot else shot['query']) + '\n'
        if db_id and question:
            prompt += 'Given the database schema:\n'
            prompt += self.db_prompts[db_id][c_num] + '\n'
            prompt += 'Translate the natural utterance into the SQL query: ' + question + '\n'
            # Trigger CoT reasoning
            prompt += "Let's think step by step." if args.cot else 'SELECT'

    return prompt

Implementation-level observable details:

① Chat models pass few-shot examples through multi-turn user/assistant messages; Completion models concatenate them into a single string.

② In CoT mode, shot['cot'] (the full reasoning chain pre-generated in cot.py) is used directly as the assistant response.

③ Completion models forcibly append "Let's think step by step." (CoT) or "SELECT" (to trigger direct SQL continuation) at the end; Chat models do not need this trick because they continue directly via role=assistant.

7.3 Database schema string templates — how the five formats are generated

All formats are built and cached in self.db_prompts[db_id][c_num] by PromptMaker.__init__ at startup. The key branches are:

# util/prompt.py · __init__ simplified
for i in range(len(tabs)):
    if args.api_doc:
        # ===== Table(Column) format: OpenAI official docs style =====
        self.db_prompts[db_id][c_num] += f"# {tabs[i]}({', '.join([col[1] for col in cols if col[0] == i])})\n"
    else:
        # ===== Create(...) family: SQL DDL style =====
        self.db_prompts[db_id][c_num] += f'create table {tabs[i]} (\n'
        for j in range(len(cols)):
            if cols[j][0] == i:
                self.db_prompts[db_id][c_num] += f"    {cols[j][1]} {db['column_types'][j]}"
                if args.pf == 'eoc':  # End of Column: PK/FK at end of each column
                    if j in db['primary_keys']:
                        self.db_prompts[db_id][c_num] += ' primary key'
                    for fk in db['foreign_keys']:
                        if fk[0] == j:
                            self.db_prompts[db_id][c_num] += f' references {tabs[cols[fk[1]][0]]}({cols[fk[1]][1]})'
                self.db_prompts[db_id][c_num] += ',\n'
        if args.pf == 'eot':  # End of Table: PK/FK at end of whole table
            pks = [cols[pk][1] for pk in db['primary_keys'] if cols[pk][0] == i]
            if len(pks) > 0:
                self.db_prompts[db_id][c_num] += f"    primary key ({', '.join(pks)}),\n"
            for fk in db['foreign_keys']:
                if cols[fk[0]][0] == i:
                    self.db_prompts[db_id][c_num] += f'    foreign key ({cols[fk[0]][1]}) references ...,\n'
        self.db_prompts[db_id][c_num] = self.db_prompts[db_id][c_num][:-2] + '\n)\n'

    # ===== Append DB example rows (c_num rows, paper uses 3) =====
    if c_num > 0 and os.path.exists(db_path):
        conn = sqlite3.connect(db_path)
        cursor = conn.cursor()
        db_contents = cursor.execute(f'SELECT * FROM {tabs[i]} LIMIT {c_num}').fetchall()
        self.db_prompts[db_id][c_num] += '/*\n'
        self.db_prompts[db_id][c_num] += f"{len(db_contents)} example rows from table {tabs[i]}:\n"
        self.db_prompts[db_id][c_num] += '\t'.join([col[1] for col in cols if col[0] == i]) + '\n'
        for record in db_contents:
            self.db_prompts[db_id][c_num] += '\t'.join([str(record[col[1]]) for col in cols if col[0] == i]) + '\n'
        self.db_prompts[db_id][c_num] += '*/\n'

7.4 ACT-SQL Auto-CoT Generation — core logic in cot.py

This section operationalizes the paper's algorithm for "automatically generating chain-of-thought": given training examples (schema + question + gold SQL), it finds the most relevant question slices for every table/column/value in the SQL and concatenates them into CoT.

# cot.py · core loop (comments simplified)
for i, example in enumerate(dataset):
    if 'cot' in example: continue

    # ① Extract every contiguous (i,j) slice of the question and encode
    words = word_tokenize(example['question'])
    phrases = [' '.join(words[a:b]) for a,b in combinations(range(len(words)+1), 2)]
    phrase_encodings = sentence_encoder.encode(phrases, normalize_embeddings=True, ...)

    # ② Extract tables / columns / values referenced by the gold SQL
    tables  = get_tables_in_sql (example['sql'], dbs[example['db_id']])
    columns = get_columns_in_sql(example['sql'], dbs[example['db_id']])
    values  = get_values_in_sql (example['sql'])

    # ③ Tables already covered by their columns are not listed again (paper notes this)
    schema_items = []
    for table in tables:
        for column in columns:
            if table in column: break
        else:
            schema_items.append((table, 'table'))
    for column in columns:
        schema_items.append((column, 'column'))

    # ④ For each schema item, find the highest-similarity question slice
    schema_linkings = {}
    for schema_item in schema_items:
        encoding = sentence_encoder.encode(schema_item[0], ...)
        scores = util.cos_sim(encoding, phrase_encodings).squeeze(0).tolist()
        phrase = phrases[max(enumerate(scores), key=lambda x: x[1])[0]]
        if phrase not in schema_linkings:
            schema_linkings[phrase] = {'table': [], 'column': []}
        schema_linkings[phrase][schema_item[1]].append(schema_item[0])

    # ⑤ Assemble the CoT string (this is the format of the assistant response in the prompt!)
    example['cot'] = "Let's think step by step.\n"
    for phrase in schema_linkings:
        example['cot'] += f'According to "{phrase}",'
        if schema_linkings[phrase]['table']:
            example['cot'] += f' tables [{", ".join(schema_linkings[phrase]["table"])}]'
        if schema_linkings[phrase]['column']:
            if example['cot'].endswith(']'):
                example['cot'] += ' and'
            example['cot'] += f' columns [{", ".join(schema_linkings[phrase]["column"])}]'
        example['cot'] += ' may be used.\n'
    if values:
        example['cot'] += f'Values [{", ".join(values)}] may be used.\n'
    example['cot'] += 'So the final answer is:\n'
    example['cot'] += ' '.join(example['query'].strip('\t ;').split())

The CoT template has only these four sentence patterns:

• Let's think step by step.

• According to "{phrase}", tables [...] and columns [...] may be used.

• Values [...] may be used. (if the SQL contains values)

• So the final answer is: {complete SQL}

No LLM participates in generating the CoT — the whole segment is a string assembled purely from rules + sentence embeddings. This is the key to the cost savings of "1 API call": the CoT is pre-generated offline before training, stored in the cot field of train.json, and pasted directly as an assistant exemplar at inference time.

7.5 Multi-turn Question Rewriting — get_prompt_remove_dependency()

For SParC / CoSQL, the two-stage method "flattens" multi-turn dialogues into single-turn questions. The first stage uses this prompt:

# util/prompt.py · get_prompt_remove_dependency()
@staticmethod
def get_prompt_remove_dependency(gpt, questions, shots):
    def preprocess(q_list):
        result = ''
        for i, q in enumerate(q_list):
            result += str(i + 1) + '. ' + q + '\n'
        return result.strip()

    if gpt in GPT_CHAT_MODELS:
        prompt = [{'role': 'system',
                   'content': 'Given the list of questions, you need to '
                              'rewrite them to remove the context dependency.'}]
        for shot in shots:  # paper: 10 manually-annotated rewrite examples per dataset
            prompt.append({'role': 'user',      'content': preprocess(shot['q_multiturn'])})
            prompt.append({'role': 'assistant', 'content': preprocess(shot['q'])})
        prompt.append({'role': 'user', 'content': preprocess(questions)})
    return prompt

The original multi-turn questions are numbered as "1. ... \n2. ... \n3. ..." and sent to the LLM, which is expected to output the same numbering but rewritten as standalone questions without context dependencies. This is where errors occur (paper Table 10 case) — the LLM occasionally fails to remove dependencies or drops information.

7.6 Complete zero-shot prompt (what is actually sent to the API)

Putting all the pieces above together, the following is the message array that GPT-3.5-turbo sees under zero-shot + Create(EoT) + 3 example rows:

[
  {
    "role": "system",
    "content": "Given the database schema, you need to translate the question into the SQL query."
  },
  {
    "role": "user",
    "content": "Database schema:\ncreate table stadium (\n    Stadium_ID number,\n    Location text,\n    Name text,\n    ...\n    primary key (Stadium_ID)\n)\n/*\n3 example rows from table stadium:\nStadium_ID  Location      Name           ...\n1           Raith Rovers  Stark's Park   ...\n2           Ayr United    Somerset Park  ...\n3           East Fife     Bayview Stadium...\n*/\ncreate table singer (\n    ...\n)\n...\nQuestion: How many singers do we have?"
  }
]

In few-shot CoT mode, after system there are additionally 2(static) + 2(dynamic) = 4 sets of (user / assistant) dialogues; the assistant content in each set is exactly the "Let's think step by step. ..." string pre-generated by cot.py.

In short, ACT-SQL's prompt philosophy: it does not rely on carefully sculpted instructions or multiple API calls; instead, it spends the effort on "putting good examples in the right places" — the static+dynamic hybrid exemplars provide format references, and the automatically generated CoT provides a thinking trajectory of "schema linking → writing SQL". The entire prompt is simple enough to generate with one page of code.

SECTION 08

Limitations & Future Work

① The exemplar selection strategy is relatively simple: the numbers of static/dynamic exemplars, ns and nd, are hyperparameters and still need to be chosen manually; the hybrid strategy itself is fairly plain and leaves room for improvement.

② Robustness-variant scores are still weak: scores on some Spider robustness variants are relatively poor and merit targeted follow-up exploration.

③ Multi-turn performance is the main open problem: the two-stage "rewrite → single-turn" approach can cause cascading schema-linking errors when rewriting quality is poor. Improving LLM performance on multi-turn text-to-SQL is the challenging future work explicitly named by the authors — this paper only completes a preliminary exploration.

Conclusion in one sentence

ACT-SQL uses an "automatically generated, schema-linking-like chain-of-thought" to achieve SOTA among in-context learning methods on the Spider dev set with a single API call and zero manual annotation, while preliminarily extending the method to multi-turn scenarios — balancing performance, speed, and cost.
