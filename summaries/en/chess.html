SECTION 01

Problem Statement — Why Text-to-SQL over real databases is so hard

LLMs are already strong at translating natural-language questions into SQL on small academic databases; once they face "industrial-grade" real databases, however, their accuracy still trails humans by roughly 30%. CHESS is aimed precisely at closing this gap.

The authors summarize the practical difficulty as several intertwined challenges:

Large and messy schemas

Real databases can easily contain thousands of columns (the paper's financial-schema example has 4,337 columns). Stuffing everything into the prompt not only exceeds context and explodes cost, but also distracts the model with irrelevant columns.

Need a "data catalog + actual values"

Column names alone are insufficient. The model often cannot know what strings actually appear in a categorical column (for example, 'M' vs. 'Male'), and it may not see column descriptions.

Errors propagate down the pipeline

In a multi-stage pipeline, if schema selection picks a wrong column early, the downstream SQL can collapse entirely — error propagation is a major accuracy killer.

Cost and privacy

Leading methods often make hundreds of LLM calls for a single question and inject the full schema; in production this is expensive, hard to deploy, and raises privacy concerns when the entire database is sent to proprietary APIs.

Core thesis

"Contextual Harnessing" = taming context.Each subtask feeds the LLM only the minimal yet sufficient context — the right columns, values, and descriptions — rather than dumping the whole database into it.

The paper's concrete contributions are: decomposing the task into four specialized agents (IR / SS / CG / UT), each mapped to one challenge; proposing hierarchical retrieval that combines keyword extraction + LSH + vector databases; designing a selector that can adaptively prune schema based on question complexity; fine-tuning open-source DeepSeek with injected noise to resist upstream errors; and releasing the system openly with an emphasis on data privacy.

SECTION 02

Methodology — Four specialized agents

CHESS is a multi-agent workflow. Each agent contains several tools; the agent decides which tools to call and in what order. The figure below shows the default pipeline.

Figure 1 · The four-agent CHESS pipeline: IR finds context → SS prunes schema → CG generates / revises → UT creates unit tests to select the best candidate (redrawn from Figure 1 of the paper)

① Information Retriever (IR) — Finding the "right context"

IR maps clues in the question to values and descriptions that actually exist in the database, using three tools:

Keyword Extraction

A few-shot prompt asks the LLM to extract keywords, keyphrases, and named entities from the question and hint, outputting them as a Python list.

Entity Retrieval

The keywords are matched against "values actually stored in the database". CHESS first uses Locality Sensitive Hashing (LSH) for approximate nearest-neighbor retrieval, then filters with edit distance (syntactic similarity) and embeddings (semantic similarity). The paper reports reducing per-question value retrieval from ~5 minutes to ~5 seconds.

Context Retrieval

Descriptions of columns / tables (the data catalog) are semantically retrieved from a vector database (ChromaDB), returning the descriptions most relevant to the question.

② Schema Selector (SS) — Pruning thousands of columns into a minimal set

SS applies three progressive filtering steps, preserving primary keys / foreign keys at every step (so later JOINs do not break):

Filter Column

Treat "whether each column is relevant to the question" as a binary classification problem. Columns are evaluated independently and can be massively parallelized, cutting away a large number of irrelevant columns first.

Select Tables

Under the global schema context, chain-of-thought reasoning evaluates whether each table is necessary.

Select Columns

The schema is further narrowed to the minimal column set genuinely needed to generate SQL, with a rationale for each selected column.

Why this design:the precision of schema selection is the bottleneck for end-to-end accuracy — missing one column (low recall) can make the question unsolvable, while adding too many columns (low precision) distracts generation. The three-step method keeps recall high while steadily improving precision (see §05 Table 5).

③ Candidate Generator (CG) — Generating and self-revising

  • Candidate generation:feed the pruned schema + retrieved values + descriptions into an LLM with a CoT prompt to produce SQL. It can generate a single candidate, or sample multiple candidates (up to 20) in a high-compute setting.
  • Revise:feed the candidate SQL back together with its execution result, and correct common errors according to a set of "DBA rules" (see §06). At most 3 revision rounds are used.
  • Optional fine-tuned model:fine-tune DeepSeek Coder-33B on roughly 9,500 BIRD training examples with deliberately injected noise (wrong columns with similar names), making it more robust to upstream schema-selection errors.

④ Unit Tester (UT) — Using LLM-written natural-language unit tests to pick the best candidate

When CG produces multiple candidates, UT selects the most correct one:

Cluster

Cluster candidates by their "execution results" — candidates with identical results are grouped together.

Generate unit tests

Ask the LLM to generate a batch of natural-language unit tests (e.g., "the answer SQL should mention... / should use..."), and require these tests to distinguish between different candidates.

Score and vote

Each candidate is judged pass/fail for each test, then the scores are aggregated to select a winner. The benefit saturates at roughly 10 tests.

Key idea:UT does not compare against gold SQL (that would be cheating). Instead, it lets the LLM "write the exam + grade the exam". In essence, it is a form of test-guided self-consistency selection that replaces simple majority voting.

SECTION 03

Core Contributions — An architecture composable by compute and privacy constraints

The strongest selling point of CHESS is not any single agent, but the fact that these four agents can be assembled like building blocks. The paper gives two representative configurations, landing at opposite ends of the "accuracy ↔ cost" spectrum.

CHESS(IR, CG, UT) · High-compute route

Skip Schema Selector and use a strong model (Gemini-1.5-pro) to sample 20 candidates + 10 unit tests, letting UT choose the best one.

71.10%

BIRD test EX — less than 2% behind the then-leading proprietary method, while using about 83% fewer LLM calls.

CHESS(IR, SS, CG) · Low-compute / privacy route

Add Schema Selector to prune the schema, generate a single candidate + at most 3 revisions, and do not run UT. Each question uses only about 6 GPT-4-turbo calls (competitors often use ~100).

65.00% / 61.5%

BIRD dev EX: proprietary model 65.00% · fully open-source (Llama-3-70B + fine-tuned DeepSeek) 61.5%, deployable entirely locally without sending data out.

Industrial value of Schema Selector:on a real financial schema with 4,337 columns, pruning with SS before generation achieves a ×5 reduction in token usage relative to feeding the full schema directly, while also improving accuracy by +2% (Pass@1 / Pass@5 ≈ 61% / 63%). Less is more.

These are also the two sources of "Efficient" in the title: the UT route trades a small number of calls for high scores; the SS route trades schema pruning for lower token cost and deployability. Users can choose freely based on budget, model availability, and privacy requirements.

SECTION 04

Datasets & Evaluation Metrics

BIRD

12,751

question-SQL pairs across 95 large databases, totaling 33.4 GB and 37 professional domains. The main battleground.

Spider

2,147

test examples (200 DB schemas). Spider has no column descriptions, so CHESS disables context retrieval here.

SDS · Subsampled dev

147

10% subset of BIRD dev: 81 simple / 54 moderate / 12 challenging, used for low-cost ablations.

Metrics

  • Execution Accuracy (EX):the main metric — whether predicted SQL and gold SQL produce identical execution results on the database.
  • Exact-Set-Match (EM):exact clause-level matching, with high false negatives (different SQL forms with the right result are still marked wrong); not the main metric.
  • Valid Efficiency Score (VES):jointly considers correctness and execution speed; because its variance is too large, the paper does not analyze it deeply.

The prompts below are taken directly from the official repo's templates/ directory, not from the paper appendix (the two often differ; the code is authoritative). I selected four templates that best represent the style of each agent.

IR · Keyword extraction (few-shot, outputs a Python list)

extract keywords, keyphrases, and named entities. ...

Hint: "the most number of competitors refer to MAX(COUNT(person_id)); ..."
["Winter Olympics", "Summer Olympics", "1988", "1988 Summer",

Hint: {HINT}

3. When no column is specified, prefer id over name between id and name.
9. Concatenating strings with || ' ' || is forbidden — and "punishable by death".
   Never concatenate columns in the SELECT clause.
10. When joining multiple tables, use aliases T1, T2, T3 ... and reference columns via aliases.
11. When computing on / sorting by a column, remember to filter NULLs from that column.

Predicted query: {SQL}

UT · Generating natural-language unit tests that can distinguish candidates

# templates/template_generate_unit_tests.txt (excerpt)
Generate a set of {UNIT_TEST_CAP} unit tests that would evaluate
the correctness of SQL queries that would answer the question.
# Key point: each test must "distinguish at least two candidates"

- Format like 'The answer SQL query should mention...',
  '... should state...', '... should use...'
- VERY IMPORTANT: focus only on the SQL "logic"; ignore output format or values.

You are provided with different clusters of candidate responses.
# Candidates are first clustered by execution result; tests must distinguish candidates within each cluster

<Answer> ['unit test #1', 'unit test #2', ...] </Answer>

The repo also includes the paired template_evaluate.txt: feed a candidate + unit tests back into the LLM and ask it to output a [Passed] / [Failed] list for each candidate — this is the "grading the exam" step in UT.

SECTION 07

Limitations & Future Work

Still behind humans:on the hardest BIRD setting, human query-synthesis accuracy is still higher than CHESS; the authors explicitly state that future work should keep narrowing this gap.

Schema selection is the bottleneck:the authors identify "higher-precision schema selection methods" as the research direction with the highest leverage — schema selection has an outsized effect on end-to-end accuracy.

Lack of large-scale benchmarks:existing benchmarks do not sufficiently reflect the challenges of real "large databases"; BIRD itself also contains semantically ambiguous questions and incorrect gold SQL, which lowers the attainable ceiling.

UT still relies on general-purpose models:unit-test generation and grading currently use general LLMs; the authors plan to fine-tune models specifically for test generation/evaluation, making the UT route more reliable.

One-sentence takeaway

CHESS demonstrates that in Text-to-SQL, "giving the model less context, but the right context" often beats "giving more context and calling it more times". By decomposing the task into four pluggable IR / SS / CG / UT agents, users can freely trade off accuracy, cost, and privacy — rather than being forced into one expensive monolithic pipeline.
