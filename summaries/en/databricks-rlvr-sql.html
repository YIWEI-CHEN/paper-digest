SECTION 01

Problem Statement — Why Enterprises Need Custom Reasoning Models

Off-the-shelf LLMs shine on generic tasks but stumble on organization-specific knowledge. This report argues for post-training a specialized reasoning model with Reinforcement Learning from Verifiable Rewards (RLVR), using text-to-SQL as a measurable proxy task.

The paper opens by naming the recurring pain points of stock LLMs on bespoke enterprise tasks: they don't know organization-specific jargon (e.g. how a company defines "churned users"), they're unaware of internal preferences (always sort certain queries by a given column), they don't know how to use internal APIs/tools, and they don't know where the data lives. Unlike math problems, this organizational knowledge is often protected and underrepresented on the internet, so it never made it into pretraining.

One way to close this gap is post-training. For data-science and coding tasks, you can frequently compute a verifiable reward: no learned reward model needed — it is the objective quality metric. For text2sql, you simply execute the model's SQL and the gold SQL against the database and compare results, yielding a clean 0/1 correctness signal. That is exactly RLVR's sweet spot.

Core claim: No elaborate agentic pipeline, no massive curated training data, no proprietary model. A single, simple-but-complete RLVR recipe trains one open-source model to state-of-the-art text-to-SQL.

Four contributions

① SOTA on first submission

Their very first BIRD leaderboard entry takes single-model SOTA on the private test set: 73.56% single-call and 75.68% with self-consistency.

② A simple, general recipe

Model/prompt selection → TAO warm-up → online RLVR → self-consistency. No data beyond the BIRD train set, and no proprietary models.

③ Stronger generalization

dev 70.80% → test 73.56% — almost no drop from dev to the held-out private test, generalizing better than other top submissions.

④ Cheaper inference

Self-consistency uses only 7 generations to beat the runner-up (which needs 8–32), a better compute-accuracy trade-off.

A counter-intuitive finding: on BIRD, proprietary models like GPT-4o, O3, and Claude Sonnet actually underperform open models from the Qwen and Llama families. This challenges the assumption that closed models are always stronger; the authors leave a deeper investigation to future work.

SECTION 02

Methodology — Pick the Right Base, Define the Right Reward

Before any training, the authors do two cheap-but-critical things: choose the base model and prompt well, and define the verifiable reward to hug the evaluation metric as closely as possible.

2.1 Model & prompt selection

The team swept a range of open- and closed-source models with different prompts on BIRD dev. The winner: Qwen 2.5 32B Coder Instruct had the best dev performance — matching the best result reported by the OmniSQL study — so it became the RLVR base model.

For the prompt, they started from the OmniSQL prompt but made two important modifications:

  • Rewrote the database encoding so the LLM can understand the schema and data more easily.
  • Added instructions to elicit reasoning. Early on, with the raw OmniSQL prompt the model often emitted no reasoning trace at all, so extra instructions were added to force a reasoning trace.

Why the reasoning trace matters: the whole pipeline treats the LLM as a reasoning model — it generates a trace containing reasoning, and the final SQL is extracted from the trace via delimiters. Without reasoning, there's far less for RLVR to optimize.

2.2 Verifiable reward

The reward is deliberately simple and shared by both TAO and RLVR: take BIRD's 0-1 execution-match metric, then add a −1 penalty for SQL that is syntactically invalid / non-executable.

reward(y) =
   +1  if execute(y) == execute(gold)
    0  if it runs but the result differs
   −1  if y is syntactically incorrect / non-executable SQL
y = model-generated SQL; gold = ground-truth SQL; reward = the eval metric plus a −1 penalty term

Notably, the authors tried reward shaping (fancier reward designs) but saw no significant improvement in preliminary experiments, so they kept this minimal form — echoing the RLVR principle that the reward should track the true evaluation target as tightly as possible.

SECTION 03

Training Pipeline — TAO Warm-up + Online RLVR

This is the paper's core recipe: warm-start the model with offline RL (TAO) to give it a good inductive bias, refine it with online RLVR, then add self-consistency at inference. The three stages are complementary.

Figure 1 · RLVR fine-tuning pipeline (redrawn from the paper's Figure 1 and Section 3)

TAO offline warm-up (Test-time Adaptive Optimization)

TAO is an offline RL approach pioneered at Databricks, here used to "warm start" RLVR by instilling a good inductive bias. One TAO iteration: generate several responses per BIRD training point with the current model → score each with the verifiable reward → run offline RL optimization on the collected data. The procedure can be repeated for multiple rounds.

Online RLVR (on top of the best TAO model)

Online RLVR via the Databricks fine-tuning service. Hyperparameter sweeps showed that a lower learning rate and a lower KL-divergence coefficient are desirable; in fact, removing the KL term entirely barely hurt performance, echoing recent literature. The service builds on GRPO with several improvements — focusing training on challenging problems and avoiding reward saturation.

Inference-time self-consistency

On the fine-tuned model, generate 7 responses and pick the most consistent answer via a parameter-free weighted majority vote. The gains from RLVR and self-consistency are complementary.

An engineering trade-off: the authors deliberately do only limited TAO training. Preliminary experiments showed that over-training with TAO actually reduces the overall gains in the later online RL stage — so warm-up should be moderate, leaving headroom for online RLVR.

SECTION 04

Dataset — The BIRD Benchmark

All training and evaluation use a single text-to-SQL benchmark, BIRD, with no extra data added on purpose — to show that the RL is doing the heavy lifting, not data engineering.

Dev

1,534

Test

1,789

private, not public

Metric

0/1

execution-result match

Each BIRD datapoint contains: a natural-language user query, an evidence field with extra instructions, a database with multiple tables, and a gold SQL as ground truth. The authors merge query and evidence into a single unified user query fed to the model.

The metric is a strict 0-1 execution match: both gold and generated SQL are executed on the database; matching outputs score 1, otherwise 0. The test set is not public, which avoids data-contamination and benchmark-hacking concerns — and is precisely why "almost no drop from dev to test" is so convincing.

Why text2sql is hard: the challenge is understanding the query in the context of a complex database — many tables, many columns, ambiguous column names, correct cell values that must be found in the data, and tables that may need to be joined to answer the question.

SECTION 06

Worked Examples — What the Model Is Thinking

The appendix lists a few sample generations. The most interesting show the model "write a first version, then self-optimize," plus a textbook failure mode. Excerpted from the paper's appendix.

Example A (success): the model self-corrects, optimizing away a JOIN

Question: Among the superheroes with the race god/eternal, how many are male?
Evidence: race "god/eternal" refers to race_id = 21; male refers to gender.id = 1. Database: superhero

The model first writes a literal version joining three tables:

SELECT COUNT(*)
FROM superhero AS h
JOIN race AS r ON h.race_id = r.id
JOIN gender AS g ON h.gender_id = g.id
WHERE r.race = 'God/Eternal' AND g.gender = 'Male';

Then it reads the evidence, realizes it can use race_id = 21 directly to drop the join on the race table, and rewrites a leaner version on its own:

-- use the race_id from the evidence to filter directly, dropping a JOIN
SELECT COUNT(*)
FROM superhero AS h
JOIN gender AS g ON h.gender_id = g.id
WHERE h.race_id = 21 AND g.gender = 'Male';

Why this example is representative: it shows the "reason → self-correct" behavior RLVR trains for — the model doesn't just translate, it loops back to exploit the evidence and make the query more robust.

Example B (failure): wrong table + missing DISTINCT

Question: What's the German type of the card "Ancestor's Chosen"? Database: card_games

SELECT fd.type
FROM cards c
JOIN foreign_data fd ON c.uuid = fd.uuid
WHERE c.name = 'Ancestor''s Chosen' AND fd.language = 'German';

The paper flags two errors: (1) the model selects type from the foreign_data table rather than the cards table — the names are confusingly similar but hold different values; (2) it forgot to apply DISTINCT to the returned values. This is a classic text2sql trap under "ambiguous column names + multi-table context."

SECTION 07

Limitations & Future Work

This is a technical report positioned to showcase Databricks' RLVR/TAO fine-tuning services, so it holds back some implementation details (the service-internal RLVR improvements). Below are the limitations and open questions the authors explicitly call out.

Limitation 1 — BIRD is only a proxy: the authors repeatedly stress that BIRD is a proxy for enterprise tasks. The real value claim is "the recipe is simple, so it's broadly applicable to BI, data science, and coding," but the report itself only validates on BIRD.

Limitation 2 — why proprietary models lag is unexplained: "proprietary LLMs underperform Qwen/Llama on this task" is a counter-intuitive observation, and the authors explicitly leave a deeper investigation to future work without offering a mechanistic explanation.

Implementation details withheld: the RLVR service's improvements over GRPO ("focus on hard problems, avoid reward saturation") are described only conceptually; TAO's exact optimization objective and hyperparameter sweep ranges aren't fully disclosed. There is no public code repository — the method will roll into Databricks' Agent Bricks product.

In one sentence

The report's message is crisp: on tasks with a computable verifiable reward, rather than stacking complex agentic pipelines or massive curated data, nail the recipe of "right base + simple reward + TAO warm-up + online RLVR + lightweight self-consistency" — a single open-source 32B model can take text-to-SQL SOTA, with cheaper compute and steadier generalization.
