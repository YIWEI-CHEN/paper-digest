SECTION 01

Problem Statement — Schema Linking as a Double-Edged Sword

Schema linking is a standard noise-reduction technique for LLM-based Text-to-SQL, but it introduces two risks at the same time: omitting required columns and damaging schema-structure completeness. RSL-SQL explicitly models that risk as a trade-off between “positive gain” and “negative impact,” then designs four steps to maximize the former while minimizing the latter.

Why schema linking can make things worse while reducing context

Putting the full database schema—table names, column names, foreign keys, comments, and sample values—into the prompt is information-complete, but it (1) pushes token usage toward the limit and (2) introduces a large amount of irrelevant noise that can steer the LLM in the wrong direction. Schema linking first selects a subset relevant to the question and only then passes it to the LLM.

The paper defines this trade-off through two risks:

Risk 1 (missed recall): if schema linking fails to recall all required tables/columns, the generated SQL must be wrong (assuming the LLM does not hallucinate nonexistent columns).

Risk 2 (structural damage): even if all required columns are recalled, schema linking can still ignore foreign-key relationships and join incorrectly, reduce columns until their semantics become more ambiguous, or amplify ambiguity—thereby turning an originally correct answer into an incorrect one.

The paper quantifies these two effects as two variables:

net gain = y − x
y = number of questions wrong with the full schema → correct after simplification (positive gain)
x = number of questions correct with the full schema → wrong after simplification (negative impact)

Schema linking pays off only when y > x. RSL-SQL’s overall strategy is to amplify y and suppress x.

The paper’s two headline contributions

Method

Proposes RSL-SQL, a four-step framework combining Bidirectional Schema Linking, Contextual Information Augmentation, Binary Selection Strategy, and Multi-Turn Self-Correction, achieving open-source SOTA with 67.21% EX on BIRD dev.

Cross-model transferability

Even the cheaper DeepSeek backbone outperforms most GPT-4-based methods. GPT-4 costs 215× more per token than DeepSeek, yet the performance gap narrows substantially.

SECTION 02

Methodology — Four-Step Overview

RSL-SQL decomposes the pipeline into four LLM-call stages, with each step targeting one side of the “positive gain / negative impact” trade-off. The key design choice is to always keep a SQL₁ generated from the full schema as insurance, then let the LLM arbitrate between the “full” and “simplified” versions at the end.

Figure 1 · Four-step overview of the RSL-SQL framework (redrawn from Fig. 1 in the paper)

Shared prompt elements

All steps share the same set of prompt building blocks. The differences are whether they use the full schema 𝒮 or the simplified schema 𝒮′, and whether they include the additional contextual augmentation HAug:

𝒮 / 𝒮′

Schema: table names, column names, and foreign keys. The full version 𝒮 is used as insurance; the simplified version 𝒮′ is used from Step 2 onward.

𝒱 / 𝒱′

Value samples: a few rows sampled from each table as value examples, truncated when too long.

𝒟 / 𝒟′

Schema descriptions: semantic descriptions generated by an LLM (following TA-SQL) for each table/column. Used only after simplification, because the original full version would be too long.

ℰ

Few-shot examples: top-k similar questions selected from the training set by Euclidean distance (following DAIL-SQL/PET-SQL).

𝒬

User question — the primary input.

𝒞

Additional context (optional): BIRD’s evidence / hint field belongs to this category.

SECTION 03

STEP 1 — Bidirectional Schema Linking

The selling point of BSL is 94% strict recall (SOTA on BIRD) while reducing the average number of input columns per question by about 83%. The key is to split schema linking into two complementary directions—forward and backward—and then take their union.

Forward Schema Linking (FSL)

Use 𝒮 + 𝒱 + 𝒬 + 𝒞 to ask the LLM directly: “Which tables/columns are relevant to this question?” The required output format is table_name and table_name.column_name. If 𝒞 mentions any column names, those are added as well. The result is denoted Lfwd.

Preliminary SQL Generation

Use the full 𝒮 + Lfwd + the other elements to have the LLM generate an initial SQL₁. This step deliberately excludes 𝒟 (descriptions), because descriptions for every table/column would explode prompt length.

Backward Schema Linking (BSL')

Reverse the direction: parse SQL₁, scan every table_name.column_name in the database, and add it to Lbwd if column_name appears in SQL₁. The paper intentionally chooses “exact column-name match” rather than rigorous SQLGlot parsing. This recalls some redundant columns, but it is more robust when SQL₁ itself is wrong.

Schema Simplification

Take Lfwd ∪ Lbwd and slice 𝒮, 𝒱, and 𝒟 into the simplified 𝒮′, 𝒱′, and 𝒟′, which are passed to downstream steps.

Why bidirectional > one-way?FSL has relatively low recall (semantic grounding is not always sufficient), but BSL' relies on the assumption that “if SQL₁ is correct, then the columns it uses are the ground truth.” That assumption pushes BSL' recall to ~90%. Taking the union fills FSL’s gaps while also correcting BSL' bias when SQL₁ is wrong.

Exact column-name match vs SQLGlot: the authors’ trade-off

# Two implementations of backward schema linking

# Option A: exact column-name match (used by the paper)
for table, col in all_db_columns:
    if col in SQL_1:        # string match; will over-recall same-named columns
        L_bwd.add((table, col))

# Option B: SQLGlot AST parse (more precise but more brittle)
import sqlglot
for tbl_col in sqlglot.parse_one(SQL_1).find_all(sqlglot.exp.Column):
    L_bwd.add(tbl_col)

The authors’ judgment: Option A is noisier, but it is more fault-tolerant when “SQL₁ is already wrong”—and the remaining noise can still be filtered later by binary selection.

SECTION 04

STEP 2 — Contextual Information Augmentation

Before generating SQL₂ over the simplified 𝒮′, the LLM first “decomposes” the SQL needed for the question: it predicts potentially useful elements, conditions, and keywords, then feeds them as additional context into SQL₂ generation. Empirically, this step contributes a 2–3% EX improvement.

SQL Components Generation

Feed 𝒬, 𝒮′, 𝒱′, 𝒟′, and 𝒞 to the LLM, asking it to output three components:

HE — Elements

A list of tables / columns that may be needed, using exactly the same output format as FSL.

HC — Conditions

After decomposing the question, list potential WHERE-clause conditions.

HK — Keywords

Infer SQL keywords from indicator words in the question (e.g., MIN, DISTINCT, GROUP BY).

HAug = { 𝒟′ , HE , HC , HK }
SQL₂ = fLLM( 𝒮′ , 𝒱′ , HAug , ℰ , 𝒬 , 𝒞 )

Why is augmentation more effective than directly generating SQL?Based on a qualitative analysis of 20 sampled questions, the paper’s explanation is that after schema simplification, LLMs often “lack sufficient focus.” Decomposing zero-shot components forces the LLM to perform a preliminary “self-plan,” surfacing the relevant keywords and conditions in the prompt before SQL generation, which reduces omissions.

Example: how a keyword hint saves a question

Question: What is the preferred foot when attacking of the player with the lowest potential?

HAug (CIA output):

Elements:
[player_attributes.potential]
[player_attributes.preferred_foot]

Keywords: ['MIN', '=']

Conditions:
['preferred foot when attacking']
['player with the lowest potential']

SQL₁ (Step 1, incorrect):

SELECT preferred_foot
FROM Player_Attributes
ORDER BY potential ASC
LIMIT 1     -- misses the "ties for lowest" case

SQL₂ (Step 2, correct):

SELECT preferred_foot
FROM player_attributes
WHERE potential = (
    SELECT MIN(potential)
    FROM player_attributes
)

Once the keyword MIN is made explicit, the LLM recognizes that the lowest potential may not be unique and rewrites the query into a subquery form, avoiding the bias introduced by LIMIT 1.

SECTION 05

STEP 3 — Binary Selection Strategy

This step is the framework’s actual “risk hedge”: the LLM chooses between SQL₁ (full schema) and SQL₂ (simplified + augmented schema). It contributes about a 1.5% EX improvement and clearly reduces the negative impact introduced by Step 2.

Core insight: the full schema and simplified schema have different biases. The former preserves structure but contains noise; the latter is cleaner but may omit information. This is cheaper than voting / self-consistency because it compares only two candidates.

Decision signal: execution results, not syntax

SQL₁ and SQL₂ are executed first, producing result sets ℛ₁ and ℛ₂. The prompt given to the LLM contains:

SQL₃ = fLLM( 𝒮′ , 𝒱′ , 𝒟′ , ℰ , 𝒬 , 𝒞 , SQL₁ , SQL₂ , ℛ₁ , ℛ₂ )

Showing the actual execution results to the LLM effectively merges “semantic checking” and “execution checking” into one call—the side with an empty result or wrong shape is easy to discard.

Example: BSS rolls a wrong SQL₂ back to SQL₁

Question: Is molecule TR151 carcinogenic?

✓ SQL₁ (Step 1)

SELECT label
FROM molecule
WHERE molecule_id = 'TR151'

Execution result: [('-',)] — 1 row, answer “no”

✗ SQL₂ (Step 2)

SELECT label
FROM molecule
WHERE molecule_id = 'TR151'
  AND label = '+'

Execution result: empty — Step 2 added an unnecessary filter

After comparing the two, the LLM selects SQL₁ back as SQL₃. BSS can correct the error because the empty set ℛ₂ is a very salient fact, far more reliable than asking the LLM to compare semantics alone.

SECTION 06

STEP 4 — Multi-Turn Self-Correction

The final safety net: perform multi-turn conversational correction for high-risk SQL, such as queries that fail execution or return empty results. Empirically this step only improves accuracy modestly, but it matters for robustness because it directly allows the system to “reject” obviously broken outputs.

Rules for identifying high-risk outputs

SQL₃ is executed directly:

  • Syntax error → enter the correction loop
  • Empty result → enter the correction loop (possibly over-constrained or using the wrong column)
  • Normal result → return directly

Dialogue loop

Feed the current SQL and the error message back to the LLM and ask it to rewrite. If the rewritten query still has an error, keep the full conversation history and continue until success or the maximum number of turns N is reached:

SQL4(i+1) = fLLM( 𝒮′ , 𝒱′ , 𝒟′ , ℰ , 𝒬 , 𝒞 , SQL4(≤i) , E(≤i) )
until SQL4(i) executes successfully, or i = N 
def mtsc(SQL_3, N):
    sql, err = SQL_3, check_error(SQL_3)
    history = []
    for i in range(N):
        if err is None:
            return sql           # executes successfully and non-empty
        history.append((sql, err))
        sql = LLM_refine(history, S_prime, V_prime, D_prime, E, Q, C)
        err = check_error(sql)
    return sql                   # rounds exhausted; return current best
SECTION 07

Datasets & Metrics

BIRD

A large cross-domain benchmark emphasizing noisy external knowledge, noisy values, and complex queries—harder than Spider. This is the main evaluation setting.

Spider

A large cross-domain benchmark with complex multi-table queries, used to test generalization.

SDS

Subsampled Development Set — follows CHESS by sampling 10% of BIRD dev from each DB for ablation, keeping cost manageable.

Four evaluation metrics

EX — Execution Accuracy

The proportion of predicted SQL queries whose execution results exactly match the ground truth. This is the standard metric.

VES — Valid Efficiency Score

An execution-efficiency score for valid SQL queries, i.e., those with matching results. Provided by BIRD.

NSR — Non-Strict Recall

Total number of intersecting elements between linked schema and GT schema / total number of GT elements. Allows partial omissions.

SRR — Strict Recall Rate

The linked schema must fully contain the GT schema to score 1; otherwise 0. This is the paper’s key breakthrough point.

NSR = Σ |Sgt,i ∩ S̃i|  /  Σ |Sgt,i|
SRR = Σ 𝕀( S̃i ⊇ Sgt,i ) / n

Paper objective: maximize SRR while keeping |S̃| as small as possible—complete recall with a short prompt.

Schema Linking Recall (Table III Summary)

The paper’s core numbers: Bidirectional Schema Linking reaches 94% SRR on BIRD (GPT-4o), surpassing:

  • CHESS — 89.7% SRR (using retrieval + multi-turn LLM filtering)
  • MCS-SQL — 89.8% SRR (multi-prompt + multiple-choice)

More importantly, RSL-SQL needs only 1–2 LLM calls, whereas CHESS / MCS-SQL require multiple turns. At the same time, it uses about 83% fewer columns than the full schema. FSL alone has lower recall; BSL' alone is already close to 90%; only their union reaches 94%.

SECTION 09

Ablation — Which Components Matter Most?

The authors add components incrementally on BIRD dev and observe changes in EX. The conclusion: Step 2 (CIA) and Step 3 (BSS) are the core; together they account for more than 70% of the total gain.

Trend from incrementally adding components (BIRD dev)

Figure 2 · Illustrative trend of BIRD dev EX as steps are accumulated (redrawn from Table IV / Fig. 8 in the paper; values are endpoints mentioned in the paper text)

The paper explicitly states:

  • Basic prompt (GPT-4o) → 62.06%; DeepSeek → 57.63%
  • Each step adds 1–3 percentage points
  • Step 2 (CIA) + Step 3 (BSS) together contribute 70%+ of the total gain
  • Step 4 (MTSC) yields a smaller gain but is important for robustness

Negative impact vs Positive gain (Fig. 7)

The paper uses two bar charts to show that after Step 2 adds HAug, negative impact rises slightly while positive gain rises substantially, making the net effect positive. Step 3 (BSS) then significantly pushes negative impact back down while positive gain continues to increase.

How BSL differs across backbones

Strong and weak models respond differently to schema linking:

• GPT-4o: bidirectional > backward > forward, because a stronger model is less sensitive to noise and the recall improvement translates directly into EX gains.

• DeepSeek: backward-only schema linking is slightly better than bidirectional, because weaker models are more precision-sensitive and excessive recall becomes interference.

This echoes the conclusion of paper [40]: the stronger the model, the more recall matters; the weaker the model, the more precision matters.

Contributions of CIA sub-components (Table VI/VII)

Removing HE, HC, HK, and 𝒟′ individually shows that:

  • 𝒟′ (column descriptions) is the most stable single component—bringing 1%+ gains for both GPT-4o and DeepSeek
  • SQL components (HE+HC+HK) add another 2%+ overall, but only 0.46% on DeepSeek (because the components generated by DeepSeek itself are lower quality)
SECTION 10

Case Studies — Two Failure-Repair Modes

(A) Maximize positive gain (CIA recovers a question)

Section §04 already showed the preferred_foot + MIN potential example: after CIA predicts 'MIN' as a keyword, the LLM rewrites ORDER BY + LIMIT 1 into the correct subquery, avoiding the “tied for lowest” corner case.

(B) Minimize negative impact (BSS falls back to SQL₁)

Section §05 already showed the molecule TR151 example: Step 2 added the unnecessary filter label='+', causing ℛ₂ to be empty. BSS sees that ℛ₂ is empty while ℛ₁ returns one row '-', and correctly selects SQL₁ back.

Putting the two cases together: CIA is “offense”—adding back what should be included—while BSS is “defense”—rolling back what went too far. The cleverness of RSL-SQL is that both directions are placed in the same pipeline and compensate for each other.

SECTION 11

Limitations & Takeaways

Limitations not stated explicitly in the paper but worth noting

Limitation 1 · Backward linking depends on SQL₁ not being too poor.Exact column-name matching partially mitigates this, but if SQL₁ seriously hallucinates nonexistent column names, BSL' cannot rescue it.

Limitation 2 · BSS relies on the LLM correctly interpreting execution results.When ℛ₁ and ℛ₂ are both non-empty and both “look plausible but are wrong,” BSS has no oracle signal available.

Limitation 3 · Transferability to weaker models has a ceiling.The ablation shows that on DeepSeek, bidirectional is slightly worse than backward-only, implying that this framework may need reweighting on weaker open-source models.

Reusable design patterns in practice

Treat SQL₁ as insurance

Do not treat the first-round prompt as a disposable draft. A “draft SQL generated from the full schema” is a very cheap safety net that can cover for schema-linking failures.

Use execution results for ensembling

Cheaper and more reliable than purely semantic LLM voting. There is no need to sample N candidates; two meaningfully different candidates are enough to hedge.

Have the LLM predict keywords first

CIA’s HK effectively forces the LLM to self-plan, grounding the raw question better into SQL structure.

Distinguish schema-linking strategies for strong vs weak backbones

Do not assume “higher recall is always better”—for 7B/13B-scale models, excessive noise can instead drag down EX.
