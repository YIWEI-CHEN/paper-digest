E-SQL

Use "question reformulation" for direct schema linking: put database tables, columns, values, and conditions directly into the original question instead of filtering the schema, reaching 66.29% EX on the BIRD test set.

SECTION 01

Problem Statement

The goal of Text-to-SQL is to translate natural-language questions into executable SQL, giving non-experts a natural-language interface to databases. Even after LLMs, there remains roughly a 20% gap from human accuracy, which means state-of-the-art pipelines are still not ready for reliable real deployment.

The authors decompose the problem into three concrete challenges:

Complex schemas

Real databases have many tables and columns, making it easy for LLMs to select the wrong ones.

Ambiguous user semantics

Terms in the question may not match database items directly (for example, "Fresno" vs "Fresno County Office of Education").

Complex SQL structure

Multiple JOINs, multiple conditions, and nesting make the SQL generated by LLMs unreliable.

Traditional path: Schema Filtering, which the authors argue against

The mainstream approach has been schema filtering (schema pruning): first discard tables and columns judged irrelevant to the question, then feed the reduced schema to the LLM. Representative systems include RESDSQL, DIN-SQL, C3, and CHESS. But citing Maamari et al. (2024) and their own experiments, the authors show that when using state-of-the-art LLMs (GPT-4o level), schema filtering can actually hurt performance. The reason is that advanced LLMs can already perform implicit schema linking; explicit filtering may instead remove the correct schema elements by mistake.

This paper's alternative: Question Enrichment

Do not change the schema; instead, compress the schema into the question: write relevant tables, columns, values, and SQL construction steps directly into the user's question, so the question seen by the LLM has already completed schema linking. This is paired with candidate predicates extracted from the initial candidate SQL (using LIKE to retrieve actual candidate values from the DB), giving the model room to correct wrong values or wrong columns.

Core claim: direct schema linking via question reformulation is a more reliable strategy than schema filtering, especially for complex queries.

Five Core Contributions

A new paradigm: direct schema linking through question reformulation

Database items and conditions are integrated into the natural-language question, forming a fully enriched question that guides SQL generation.

The first use of question enrichment to improve both LLM reasoning and schema linking

The authors claim this is the first Text-to-SQL work to approach the problem from the angle of "rewriting the question".

Generate candidate predicates with the LIKE operator

Tokens are extracted from the candidate SQL, and LIKE '%token%' is run on the DB to find values that actually exist as prompt augmentation.

Empirical evidence that schema filtering hurts under advanced LLMs

It is not merely unnecessary; using it lowers performance.

Small models also benefit strongly from enriched questions

DeepSeek Coder 7B improves on BIRD dev from 37.02 to 56.45 EX (+19.43) without fine-tuning.

SECTION 02

Methodology — E-SQL Pipeline Overview

E-SQL consists of four main modules, ordered as CSG → CPG → QE → SR. An additional SF module is used only for ablation comparison and is not included in the final pipeline.

Figure 1 · E-SQL pipeline (redrawn from Figure 2 of the paper; ★ marks QE, the paper's core module)

2.1 Database Description & Value Selection — Shared preprocessing

Real DB description files and data values are too large to fit entirely into the prompt. E-SQL ranks them with BM25:

Attach the 10 most relevant values for each column

From the actual values in each column, take the top-10 by BM25 relevance to the question. If the column contains NULL values, ensure NULL is also included (so the LLM does not miss IS NOT NULL).

Attach the 20 most relevant description sentences overall

Rank sentences from the DB description file with BM25 and take the 20 most relevant ones.

This is faster than RESDSQL's Longest Common Substring (LCS) method and simpler than CHESS's LSH.

SECTION 03

Details of the Four Modules

3.1 CSG — Candidate SQL Generation

The first stage generates a "candidate SQL". The point is not for it to be correct, but for it to be roughly correct: using the right tables, mostly right columns, and roughly right values, because the downstream CPG module will extract tokens from it to retrieve candidate values from the DB. The prompt design:

  • • 9-shot examples: 3 examples for each difficulty level (simple / moderate / challenging), and examples must come from a different DB than the current query to avoid leakage
  • • Examples are ordered from simple to complex
  • • The schema uses the code representation recommended by DAIL-SQL (create table syntax, not natural language)
  • • Add the 10 column values and 20 description sentences retrieved by BM25, then trigger CoT with "Let's think step by step"

3.2 CPG — Candidate Predicate Generation

This is E-SQL's second distinctive design. The authors enumerate 6 cases for LLM-generated predicates. Cases (2)–(5) are all situations where the table, column, or value is partially wrong and can be corrected by the database's actual data:

Six cases of predicate errors (using "Fresno County Office of Education" as the example)
CaseSituationCan CPG recover?
(1)Table, column, and value are all correctNot needed
(2)Table and column are correct, but the value is incomplete ('Fresno' should be 'Fresno County Office of Education')✓
(3)Table and value are correct, but the column is wrong (County Name is used but should be District Name)✓
(4)Table is correct, but both column and value are wrong✓
(5)Value is correct, but table and column are wrong✓
(6)Everything is wrong and unrelated to the question✗

Concretely, the method extracts tokens from predicates in the candidate SQL and runs the following query on the DB:

SELECT DISTINCT <COLUMN>
FROM <TABLE>
WHERE <COLUMN> LIKE '%<VALUE>%';

This produces a list of candidate predicates in the form <table>.<column> <op> <value>, which is inserted into the prompts for QE and SR.

3.3 QE — Question Enrichment (Core of the Paper)

Do not filter the schema; write the schema into the question, while also attaching an SQL construction plan as reasoning.

Given the original question + schema + DB description + DB values + candidate predicates, the LLM produces two artifacts:

Enriched Question

Write tables, columns, values, and conditions into the question using exact names, so the LLM no longer needs to perform schema linking.

Enrichment Reasoning

Use a CoT style to explain why these schema items are selected and how the SQL should be constructed; in effect, this is an SQL construction plan.

The two are concatenated into a Fully Enriched Question, which becomes the input to downstream SR.

Few-shot design: the authors manually annotate 12 examples (4 per difficulty level) and randomly choose 9 from them (3 per difficulty); the examples are again forced to come from different DBs than the current query. The entire enrichment is completed in "one prompt", with no iteration.

3.4 SR — Predicate & Error-Aware SQL Refinement

The final step aggregates everything: the fully enriched question, candidate SQL, execution error message (if any), candidate predicates, schema, and description. The LLM then decides whether to patch the candidate SQL or write a new one.

Key design choice: E-SQL does not use self-consistency / multi-choice repeated-generation voting (used by C3, DAIL-SQL, CHESS, and MCS-SQL), nor MAC-SQL's iterative refiner agent. Each E-SQL module calls the LLM only once, saving computation cost.

3.5 SF — Schema Filtering (Ablation only)

The authors implement a single-step schema filtering module (together with a Filtered Schema Correction step) for ablation experiments to show that adding SF to the pipeline actually lowers the score. The final E-SQL pipeline therefore excludes SF.

SECTION 04

Datasets & Evaluation Metrics

Datasets

BIRD

A cross-domain, realistic, complex large-scale DB benchmark. It contains 95 DBs, 37 professional domains (football, F1, blockchain, medicine, education, ...), and 12,751 text-SQL pairs. Train 9,428 / Dev 1,534 / Test 1,789 (the test set is not public). It is the main evaluation benchmark.

Spider

200 DBs, 138 domains, and 10,181 text-SQL pairs. This paper uses the test split with 2,147 pairs.

Evaluation Metrics (BIRD trio)

EX

Execution Accuracy. It checks whether the predicted SQL and the gold SQL produce the same execution result.

R-VES

Reward-based Valid Efficiency Score. It evaluates not only correctness but also efficiency, averaging over 100 runs and assigning 0 / 0.25 / 0.5 / 0.75 / 1 / 1.25 points according to the runtime ratio τ.

Soft F1

BIRD's latest metric. It tolerates small differences such as column order and missing values.

R-VES =
  1.25  if ŷ correct AND τ ≥ 2
  1.00  if ŷ correct AND 1 ≤ τ < 2
  0.75  if ŷ correct AND 0.5 ≤ τ < 1
  0.50  if ŷ correct AND 0.25 ≤ τ < 0.5
  0.25  if ŷ correct AND τ < 0.25
  0    if ŷ incorrect

τ = (ground truth run time) / (predicted SQL run time)

Experimental Setup

Main models: GPT-4o-mini (about 1/3 the cost of GPT-4o) plus GPT-4o for stronger runs. Small-model exploration: Qwen2.5 Coder 1.5B/7B and DeepSeek Coder 1.3B/7B. Temperature 0.0, top_p 1.0, max_tokens 2048. Each module uses 9-shot examples (3/3/3 across difficulty), 10 column values, and 20 description sentences.

SECTION 08

Prompt Templates — From the GitHub Source

Appendix A of the paper lists the full prompt skeletons for the four modules (the actual files are in prompt_templates/). Below are the most important design choices; see Appendix A.1-A.4 in the paper for the complete version.

8.1 The 8-step instruction in the QE prompt (core)

The Question Enrichment prompt asks the LLM to execute 8 steps in order. Each step maps precisely to a subtask of schema linking:

# Step 1 - Read the Question Carefully: identify named entities, terms, keywords;
       link the question with the schema.
# Step 2 - Analyze the Database Schema: inspect the schema using DB samples;
       identify table / column / value relevant to the question.
# Step 3 - Review the Database Column Descriptions: use column descriptions
       to understand the meaning of each column and strengthen question-schema alignment.
# Step 4 - Analyze and Observe The Database Sample Values: from DB sample values
       observe the actual data distribution of each column; find relevant items by similarity matching.
# Step 5 - Review the Evidence: use the concrete information in the evidence to point to relevant elements.
# Step 6 - Analyze the Possible SQL Conditions: analyze candidate predicates from CPG
       and align them with phrases and keywords in the question.
# Step 7 - Identify Relevant Database Components: name the tables/columns/values that will actually be used.
# Step 8 - Rewrite the Question: write the identified schema items and conditions into the question in detail,
       making the question clear, readable, and free of redundancy.

The output JSON is forced to contain two keys:

{
  "chain_of_thought_reasoning": "... detail explanation ...",
  "enriched_question": "... expanded clean question ..."
}

Note the last line, "If you do the task correctly, I will give you 1 million dollars": the classic motivational-prompt trick, and the authors really used it.

8.2 QE output example (Appendix B)

# Original Question
Among the schools with the average score in Math over 560 in the SAT test,
how many schools are directly charter-funded?

# Enriched Question
Please find the number of schools (COUNT(frpm.`School Code`)) whose
charter funding type is directly funded
(frpm.`Charter Funding Type` = 'Directly funded'), and whose
AvgScrMath larger than 560 in the SAT test
(satscores.AvgScrMath > 560). To find the schools with the charter
funding type information and average math score in SAT, frpm and
satscores tables should be joined. Apply the charter funding type
condition and average math score condition. Calculate the number of
schools using COUNT aggregate function in the SELECT statement.

The point is clear: the enriched question is already almost an SQL plan written in natural language. The downstream LLM only needs to "translate" it. This is where the term "direct schema linking" comes from.

8.3 Shared design of CSG / SR / SF

The system prompt of every module begins the same way:

### You are an excellent data scientist. You can capture the link
### between the question and corresponding database and perfectly
### generate valid SQLite SQL query to answer the question.

Shared defensive-detail instructions (repeated in every prompt):

Wrap column names in backticks

Wrap every table name and column name individually in backticks to avoid keyword conflicts.

Use ORDER BY + LIMIT 1 instead of MAX/MIN

For extrema, the prompts force ORDER BY ... LIMIT 1, especially with multi-table JOINs.

CAST and parentheses

Pay particular attention to parenthesis placement in mathematical expressions; division should use CAST to REAL.

Use IS NOT NULL cautiously

Add it only when NULL would cause division by zero or misclassification, avoiding over-defensive SQL.

8.4 The repair logic in the SR prompt

After aggregating everything, SR decides whether to "modify the candidate SQL" or "rewrite it":

# Step 3 - Analyze the Possible SQL Query: identify potential errors in the candidate
       (missing conditions, wrong functions, misused aggregations, syntax errors, unrecognized tokens, ambiguous columns).
# Step 4 - Investigate Possible Conditions and Execution Errors:
       carefully read the candidate predicates from CPG (format: <table>.<col><op><val>),
       and analyze the candidate SQL's execution error messages (if any).
# Step 5 - Finalize the SQL: combine question intent, evidence, and candidate conditions
       to decide whether to patch or rewrite the candidate.

What Yi-Wei's Dataverse work can borrow: E-SQL's core idea of using "write the schema into the question" instead of "filter the schema" resonates methodologically with injecting SQL-mode hints for LLMs on Dataverse (for example, handling DateTime/DateOnly columns). Rather than letting the LLM infer constraints on its own, make them explicit at the question level. CPG's use of LIKE '%val%' to correct predicted values from the actual DB is also directly applicable to incomplete user-written names in CRM tables such as account/contact.

SECTION 09

Limitations & Takeaways

① Constrained by hardware and cost: Almost all experiments use GPT-4o-mini, and no fine-tuning is performed. In the small-model experiments, only Qwen2.5 Coder 7B can run the full E-SQL pipeline (the other small models lack sufficient context length). "Studying schema linking that works on small LLMs" is explicitly identified as future work.

② Dev / Test performance is inconsistent: GPT-4o performs better on test than dev, while GPT-4o-mini performs worse on test than dev. Because the BIRD test set is not public, the authors cannot directly analyze the cause; they also note that repeated LLM runs can vary.

③ Prompt design is not fully explored: The paper focuses on schema linking and data augmentation, without systematically comparing different prompt templates. The authors leave broader prompt design and iterative question refinement to future work.

④ No repeated generation / self-consistency: To save cost, each module runs only once; unlike CHESS / MCS-SQL, E-SQL does not rely on multiple sampling and voting. This is a trade-off, and the authors choose the lower-cost side.

One-sentence conclusion

E-SQL flips schema linking from "filter out irrelevant items" to "write relevant items into the question". It reaches 66.29% EX on the BIRD test set and shows that, in the era of advanced LLMs, schema filtering can be harmful. This gives Text-to-SQL a new path that is cost-controlled and especially effective for complex queries.
